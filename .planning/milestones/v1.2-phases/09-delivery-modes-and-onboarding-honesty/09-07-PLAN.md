<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| schema defaults -> open registration | Seeded runtime config controls whether a new user can complete onboarding without email delivery. |
| Login TUI -> Accounts delivery API | Returning unconfirmed users request verification delivery through the domain context. |
| operator shell -> token table | Mix tasks can generate verification codes and reset URLs outside normal user flows. |
| token/code -> operator stdout | Sensitive account recovery material leaves Foglet through operator terminal output. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09G-01 | Denial of Service | `Config.Schema` defaults/seeds | mitigate | Default no-email installs to `require_email_verification=false` so open registration does not create an impossible verification dead end. |
| T-09G-02 | Tampering | `Login.start_verify_flow/2` | mitigate | Remove direct `Accounts.build_verify_code/1` call and route through `Accounts.deliver_verification_code/1` with unavailable/failure handling. |
| T-09G-03 | Information Disclosure | no-email Mix task stdout | mitigate | Label verification codes and reset URLs as operator retrieval output; print exact "no email was sent by this task" copy. |
| T-09G-04 | Spoofing | no-email retrieval workflow | mitigate | Keep retrieval in explicit operator Mix tasks only; do not add end-user TUI relay or browser workflows that could look like normal delivery. |
| T-09G-05 | Repudiation | operator retrieval | accept | Phase 9 does not add durable operator audit logs; Mix task output is explicit and tests cover generated token side effects. |
</threat_model>
