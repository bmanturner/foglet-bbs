<objective>
Two interlocking changes that prepare for the struct-field deletion in Plan 39-07:

1. **Breadcrumb migration (D-10, D-11, D-12, R3):** Each of the four screens that today supply breadcrumb data via App fields (ThreadList, PostReader, PostComposer, NewThread) emits `breadcrumb_parts: [String.t()]` on the chrome map they pass to `ScreenFrame.render/4`. Delete `BreadcrumbBar.parts_for/1`, the per-screen pattern matches at `breadcrumb_bar.ex:62-92`, the `board_name/1` and `thread_title/1` legacy field readers, and the `screen_state_for/2` private helper that supports them. Update `ScreenFrame.normalize_chrome/2` so its `Map.put_new` fallback is `["Foglet"]` instead of a `parts_for/1` call.

2. **Legacy callback body rewrites (D-21, RESEARCH Pitfall 1):** PostReader's legacy `render/1` (lines 225-243), `handle_key/2` (328-380), `load_posts/2` (400-428), `flush_read_pointers/2` (455-476), `advance_post/2` (650-686), `scroll_post/2` (688-720), `build_flush_context/1` (722-741), and `frame_state/2` (890-903) currently read `state.posts`, `state.read_position`, `state.current_thread`, `state.current_board`, and `state.composer_draft` from the App struct. PostComposer lines 400, 420, 544 similarly read `state.composer_draft` and `Map.get(state, :current_thread)`. Rewrite these BODIES — not the declarations — to source the same data from `state.screen_state[:post_reader]` / `[:post_composer]` (the `%PostReader.State{}` / `%PostComposer.State{}` structs). The transitional callback DECLARATIONS (`render/1`, `handle_key/2`, `init_screen_state/1`) remain in `screen.ex` (Phase 40 owns their removal).

Why both in one plan: the legacy callback rewrites and the breadcrumb migration are the only two changes that make Plan 39-07's struct-field deletion safe. Combining them keeps Plan 39-07 a clean "delete 7 fields" operation. They're also independent of each other AT THE FILE LEVEL within this plan — the breadcrumb work touches `render/2` and chrome maps; the legacy-body work touches different functions inside the same files. They can be done sequentially within this plan, single commit.

Per RESEARCH Pitfall 5 / Assumption A3: the `frame_state/2` plain-map keys (`current_screen:`, `current_board:`, etc.) are NOT the same as the deleted `%App{}` struct fields — they're plain map keys. Some can stay; only readers that consume them via the deleted BreadcrumbBar path get pruned.

Purpose: STATE-02 (screen-owned data not in App fields), STATE-03 (helper modules don't read App state), R2 (no production reader of deleted fields), R3 (breadcrumb chrome reads explicit input).

Output: Six files modified. Render-smoke-equivalent behavior preserved (breadcrumbs still render). `mix compile --warnings-as-errors` stays green so Plan 39-07 can land the struct deletion atomically.
</objective>
