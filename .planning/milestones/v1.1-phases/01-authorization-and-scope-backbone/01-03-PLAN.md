<objective>
Add a new actor-aware, non-bang `Foglet.Config.put/3` function that returns tagged tuples for ALL failure modes (`{:error, :forbidden}`, `{:error, :unknown_key}`, `{:error, :invalid_value}`). Do NOT modify the existing `Foglet.Config.put!/3` — seeds and 20+ test call sites depend on its raise contract and 3-arity signature.

Purpose: Phase 2's sysop TUI config editor is the first live caller that needs a clean `case`-match API for authorization rejection and validation errors. The existing `put!/3` is a trusted seed/test path that raises on invalid input; adding authorization to it would create a mixed return contract (`with :ok <- Config.put!(...)` would crash on validation). The new `put/3` isolates the interactive caller surface from the raise-based internal API.

Output: `Foglet.Config.put/3` that wraps the existing `do_put!` validation pipeline in an authorization guard and tagged-tuple error conversion, plus focused tests.
</objective>

    ```elixir
    @doc """
    Actor-aware config write (D-19). Returns tagged tuples for all failure modes; never raises.
    Use this from interactive callers (TUI sysop screen in Phase 2, future API clients).

    For trusted internal paths (seeds, Mix tasks, test setup), call `put!/3` directly with
    `nil` as `updated_by_id` — the non-actor pathway continues unchanged.

    Authorization: the actor must be permitted for `:edit_config` at `:site` scope (Foglet.Authorization).
    Validation errors (unknown key, invalid value) are returned as `{:error, :unknown_key}` /
    `{:error, :invalid_value}` instead of raising.

    On a forbidden or validation failure, no DB mutation and no ETS invalidation occur.
    """
    @spec put(Foglet.Accounts.User.t() | nil, String.t(), term()) ::
            {:ok, Entry.t()} | {:error, :forbidden} | {:error, :unknown_key} | {:error, :invalid_value}
    def put(actor, key, value) when is_binary(key) do
      with :ok <- Bodyguard.permit(Foglet.Authorization, :edit_config, actor, :site) do
        case Schema.validate(key, value) do
          :ok ->
            {:ok, do_put!(key, value, actor && actor.id)}

          {:error, {:unknown_key, ^key}} ->
            {:error, :unknown_key}

          {:error, %{reason: _reason}} ->
            {:error, :invalid_value}
        end
      end
    end
    ```

    **CRITICAL — do NOT modify `put!/3`.** Its signature, raise contract, and body stay exactly as they are today. Seeds (`priv/repo/seeds/config.exs`) and existing tests (`config_test.exs`, `login_test.exs`, `verify_test.exs`, etc.) depend on this.

    **CRITICAL Pitfall 2 reminder:** `Bodyguard.permit/4` happens at the top of the `with` block — outside any ETS invalidation and outside the `do_put!` call (which performs the DB mutation). The `with :ok <- ...` short-circuit ensures that if the authorization fails, NOTHING downstream runs.

    **Why `actor && actor.id`:** Bodyguard will reject `nil` actor before we get here, so we could technically use `actor.id` directly. The `actor && actor.id` form is defensive — if a future refactor allows nil through for some reason, we don't crash on `nil.id`.

    **Why `{:error, %{reason: _reason}}` not the full struct:** `Schema.validate/2` returns `{:error, %{reason: r, expected: e, got: g}}` for invalid values. The new `put/3` collapses this into `{:error, :invalid_value}` — Phase 2 callers don't need the per-field reason; they'll re-validate at form level. If Phase 2 later needs the full error detail, it can be added without breaking the contract (add a new return variant; existing callers still match on `{:error, :invalid_value}`).
