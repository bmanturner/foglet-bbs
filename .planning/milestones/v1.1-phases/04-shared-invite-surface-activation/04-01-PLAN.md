<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| TUI key input -> shared invite actions | Untrusted terminal events request generate/revoke/list behavior. |
| Shared invite actions -> Foglet.Accounts | TUI crosses into authoritative domain APIs for all invite mutations and listing. |
| Foglet.Accounts status maps -> terminal render | Persisted invite metadata is displayed to an actor and must not be expanded by TUI queries. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | E | InvitesActions.generate/revoke_selected | mitigate | Call only `Foglet.Accounts.create_invite/1` and `Foglet.Accounts.revoke_invite/2`; never infer authorization from tab visibility. |
| T-04-02 | I | InvitesSurface.render/2 | mitigate | Render only the status maps returned by `Accounts.list_invites/1`; no direct Repo calls or cross-actor filtering in TUI. |
| T-04-03 | T | InvitesActions error handling | mitigate | On `:forbidden`, `:limit_reached`, `:not_found`, `:unavailable`, or changeset error, set `InvitesState.error` and do not set `last_generated_code` or optimistic revoked status. |
| T-04-04 | R | Revoke result display | mitigate | Refresh via `Accounts.list_invites/1` only after `Accounts.revoke_invite/2` returns `{:ok, _}` so failed revoke cannot spoof success. |
</threat_model>
