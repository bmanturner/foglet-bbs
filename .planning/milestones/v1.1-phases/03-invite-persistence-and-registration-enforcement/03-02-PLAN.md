  <behavior>
    - Sysops can generate invites under every `invite_code_generators` policy.
    - Mods can generate invites when policy is `"mods"` and are rejected when policy is `"sysop_only"` or `"any_user"`.
    - Regular users can generate invites only when policy is `"any_user"` and Bodyguard authorization is made to permit them through an explicit domain policy decision if required by existing `Foglet.Authorization`.
    - For `"any_user"`, `invite_generation_per_user_limit() == 0` allows unlimited generated invites; positive values cap invites generated by the same `issuer_id`.
    - Generated codes use OTP `:crypto.strong_rand_bytes/1` or equivalent existing secure randomness, are uppercase alphanumeric, and are not predictable counters.
  </behavior>
  <action>
    In `Foglet.Accounts`, add alias `Invite` to `alias Foglet.Accounts.{Invite, SSHKey, User, UserToken}`. Add an invite section with `create_invite/1`, private `ensure_invite_generation_policy/1`, private `ensure_invite_generation_limit/1`, and private `generate_invite_code/0`.

    `create_invite/1` must first call `Bodyguard.permit(Foglet.Authorization, :generate_invite, actor, :site)` and return `{:error, :forbidden}` on denial. Then enforce runtime policy from `Foglet.Config.invite_code_generators/0`: `"sysop_only"` permits only `%User{role: :sysop}`, `"mods"` permits `%User{role: role}` when role in `[:mod, :sysop]`, and `"any_user"` permits active authorized users while applying cap logic. For any non-matching policy/role, return `{:error, :forbidden}`.

    Cap logic only applies when policy is `"any_user"` and actor role is `:user`: read `Foglet.Config.invite_generation_per_user_limit/0`; when the value is `0`, permit; when positive, count existing invites with `issuer_id == actor.id` and reject with `{:error, :limit_reached}` when count is greater than or equal to the cap.

    Generate codes with `:crypto.strong_rand_bytes(16)`, encode with `Base.encode32(case: :upper, padding: false)`, and strip/keep only `A-Z0-9` characters if needed. Use a retry loop around `Repo.insert/1` for unique-constraint collisions, capped at 5 attempts; return the changeset error if all retries fail. Insert with `%Invite{issuer_id: actor.id} |> Invite.changeset(%{code: code}) |> Repo.insert()`.

    If existing `Foglet.Authorization` currently denies regular users for `:generate_invite`, update the policy in the smallest possible way so regular active users may pass only for `:generate_invite` at `:site`; runtime policy in `Accounts.create_invite/1` still decides whether `"any_user"` is enabled. Do not broaden `:revoke_invite`.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Actor -> Accounts invite APIs | Untrusted caller attempts generation/revocation/status operations. |
| Accounts -> Config policy | Runtime config controls who can generate invites and cap behavior. |
| Accounts -> database | Public invite codes and lifecycle mutations are persisted. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Spoofing | `create_invite/1` code generation | mitigate | `generate_invite_code/0` uses `:crypto.strong_rand_bytes/1` and unique DB constraint; acceptance greps block predictable counters. |
| T-03-02 | Tampering | `revoke_invite/2` state mutation | mitigate | Revocation uses conditional `Repo.update_all` over available invites only; consumed invites remain unchanged. |
| T-03-03 | Elevation of privilege | `create_invite/1`, `revoke_invite/2` | mitigate | Both side-effecting APIs call `Bodyguard.permit/4` inside `Foglet.Accounts`, not in UI. |
| T-03-04 | Information disclosure | invite status/revocation errors | accept | Domain status APIs intentionally expose status to authorized/domain callers per spec; Plan 03 masks registration errors for user-entered codes. |
</threat_model>
