### Locked Decisions

- Phase 3 is domain/API only; no shared `INVITES` tab activation or TUI command wiring belongs in this phase. [VERIFIED: .planning/phases/03-invite-persistence-and-registration-enforcement/03-SPEC.md]
- Invite records are v1.1 single-use records with raw generated public `code`, `issuer_id`, `consumed_at`, `consumed_by_user_id`, `revoked_at`, and timestamps. [VERIFIED: .planning/phases/03-invite-persistence-and-registration-enforcement/03-SPEC.md]
- Derived invite status is `:available`, `:consumed`, or `:revoked`. [VERIFIED: .planning/phases/03-invite-persistence-and-registration-enforcement/03-SPEC.md]
- Invite generation must enforce `Authorization` action `:generate_invite`, `Foglet.Config.invite_code_generators/0`, and `Foglet.Config.invite_generation_per_user_limit/0` when policy is `"any_user"`. [VERIFIED: .planning/phases/03-invite-persistence-and-registration-enforcement/03-SPEC.md]
- `invite_generation_per_user_limit == 0` means unlimited; positive values cap per-user generation in `"any_user"` mode. [VERIFIED: .planning/phases/03-invite-persistence-and-registration-enforcement/03-SPEC.md]
- Revocation must enforce `Authorization` action `:revoke_invite`, set `revoked_at` only for available invites, reject missing or consumed invites, and leave consumed invites unchanged. [VERIFIED: .planning/phases/03-invite-persistence-and-registration-enforcement/03-SPEC.md]
- In `invite_only` mode, registration must reject missing, unknown, revoked, or consumed invite codes with a changeset-style `invite_code` field error. [VERIFIED: .planning/phases/03-invite-persistence-and-registration-enforcement/03-SPEC.md]
- A valid invite must be consumed only when user creation succeeds, and concurrent redemption must not create two users from one invite. [VERIFIED: .planning/phases/03-invite-persistence-and-registration-enforcement/03-SPEC.md]
- Do not add HTTP, date/time, invite hashing, expiry, notes, multi-use limits, campaign/referral trees, search/filtering, email delivery, or verification-code UX changes in this phase. [VERIFIED: .planning/phases/03-invite-persistence-and-registration-enforcement/03-SPEC.md]

## Summary

Use the existing Phoenix context pattern: add `Foglet.Accounts.Invite` plus an `invite_codes` migration, then expose `Foglet.Accounts` functions for generation, listing/status, revocation, lookup/validation, and invite-only registration redemption. [VERIFIED: lib/foglet_bbs/accounts.ex] [VERIFIED: docs/ARCHITECTURE.md] Keep the trust boundary in the domain context, because `Foglet.Authorization` explicitly says domain context functions are the real trust boundary and TUI checks are advisory only. [VERIFIED: lib/foglet_bbs/authorization.ex]

The hard part is not code generation; it is transaction shape. Registration currently builds a user changeset and inserts the user directly, then subscribes to default boards after commit. [VERIFIED: lib/foglet_bbs/accounts.ex] Phase 3 should preserve normal open registration behavior, but for `invite_only` it should validate the user changeset before consuming an invite, then use one database transaction that inserts the user and conditionally updates the invite row from available to consumed. [CITED: https://hexdocs.pm/ecto/Ecto.Multi.html] [CITED: https://www.postgresql.org/docs/16/sql-update.html]

**Primary recommendation:** Implement invites as an Accounts-domain Ecto schema and use a conditional `UPDATE ... WHERE code = ? AND consumed_at IS NULL AND revoked_at IS NULL` inside the same transaction as user insertion; do not consume invites in the registration screen preflight. [VERIFIED: lib/foglet_bbs/tui/screens/register.ex] [CITED: https://hexdocs.pm/ecto/Ecto.Multi.html] [CITED: https://www.postgresql.org/docs/16/sql-update.html]

## Architecture Patterns

### System Architecture Diagram

```text
TUI Register Screen
  |
  | submits %{handle, email, password, invite_code}
  v
Foglet.Accounts.register_user/1
  |
  +--> registration_mode != "invite_only"
  |      |
  |      v
  |   existing User.registration_changeset -> Repo.insert -> subscribe_to_defaults
  |
  +--> registration_mode == "invite_only"
         |
         v
      User.registration_changeset pre-validation
         |
         +--> invalid user attrs -> return {:error, changeset}; invite untouched
         |
         v
      Repo.transaction / Ecto.Multi
         |
         +--> insert user
         |
         +--> conditional consume invite:
              UPDATE invite_codes
              SET consumed_at = now, consumed_by_user_id = user.id
              WHERE code = input
                AND consumed_at IS NULL
                AND revoked_at IS NULL
         |
         +--> update count == 1 -> commit user + invite consumption
         |
         +--> update count == 0 -> rollback and return changeset error on :invite_code
```

### Pattern 1: Context-Owned Invite Schema

**What:** Add `Foglet.Accounts.Invite` in its own file, `use Foglet.Schema`, and map table `invite_codes` with `code`, `issuer_id`, `consumed_at`, `consumed_by_user_id`, `revoked_at`, and timestamps. [VERIFIED: docs/DATA_MODEL.md] [VERIFIED: lib/foglet_bbs/schema.ex]  
**When to use:** This phase adds persisted Accounts-domain data and should follow the existing `User`, `SSHKey`, and `UserToken` organization. [VERIFIED: lib/foglet_bbs/accounts/user.ex] [VERIFIED: lib/foglet_bbs/accounts/ssh_key.ex] [VERIFIED: lib/foglet_bbs/accounts/user_token.ex]  
**Example:**

```elixir
# Source: local schema convention in lib/foglet_bbs/accounts/user.ex
defmodule Foglet.Accounts.Invite do
  use Foglet.Schema

  schema "invite_codes" do
    field :code, :string
    field :consumed_at, :utc_datetime_usec
    field :revoked_at, :utc_datetime_usec

    belongs_to :issuer, Foglet.Accounts.User
    belongs_to :consumed_by_user, Foglet.Accounts.User

    timestamps(type: :utc_datetime_usec)
  end

  def status(%__MODULE__{revoked_at: revoked_at}) when not is_nil(revoked_at), do: :revoked
  def status(%__MODULE__{consumed_at: consumed_at}) when not is_nil(consumed_at), do: :consumed
  def status(%__MODULE__{}), do: :available
end
```

### Pattern 2: Database Constraint First

**What:** Create a unique index on `invite_codes.code`, foreign keys to `users`, and indexes for issuer/status review. [VERIFIED: .planning/phases/03-invite-persistence-and-registration-enforcement/03-SPEC.md]  
**When to use:** Use database constraints for uniqueness, then annotate changesets with `unique_constraint/3` so Ecto converts DB violations into changeset errors. [CITED: https://hexdocs.pm/ecto/Ecto.Changeset.html]  
**Example:**

```elixir
# Source: Ecto.Changeset unique_constraint docs and local migration conventions
create table(:invite_codes, primary_key: false) do
  add :id, :uuid, primary_key: true
  add :code, :string, null: false
  add :issuer_id, references(:users, type: :uuid, on_delete: :restrict), null: false
  add :consumed_at, :utc_datetime_usec
  add :consumed_by_user_id, references(:users, type: :uuid, on_delete: :nilify_all)
  add :revoked_at, :utc_datetime_usec

  timestamps(type: :utc_datetime_usec)
end

create unique_index(:invite_codes, [:code])
create index(:invite_codes, [:issuer_id, :inserted_at])
```

### Pattern 3: Conditional Update for Single-Use Redemption

**What:** Consume an invite with a conditional update that only matches available invites; treat `count == 1` as success and `count == 0` as invalid/unavailable. [CITED: https://www.postgresql.org/docs/16/sql-update.html]  
**When to use:** Use during registration transaction and revocation so unavailable invites do not mutate. [VERIFIED: .planning/phases/03-invite-persistence-and-registration-enforcement/03-SPEC.md]  
**Example:**

```elixir
# Source: Ecto.Multi update_all docs + PostgreSQL UPDATE count semantics
query =
  from i in Invite,
    where: i.code == ^code,
    where: is_nil(i.consumed_at),
    where: is_nil(i.revoked_at)

{count, _} =
  repo.update_all(query,
    set: [
      consumed_at: now,
      consumed_by_user_id: user.id,
      updated_at: now
    ]
  )

case count do
  1 -> {:ok, :consumed}
  0 -> {:error, :invalid_invite_code}
end
```

### Pattern 4: Changeset-Style Invite Errors

**What:** Build a user registration changeset and attach `invite_code` errors with `Ecto.Changeset.add_error/3` rather than raising or returning bare atoms to the TUI. [CITED: https://hexdocs.pm/ecto/Ecto.Changeset.html]  
**When to use:** Missing, unknown, revoked, consumed, and race-lost redemption must all return `{:error, changeset}` with `invite_code` in `changeset.errors`. [VERIFIED: .planning/phases/03-invite-persistence-and-registration-enforcement/03-SPEC.md]  
**Example:**

```elixir
# Source: Ecto.Changeset.add_error docs
defp invite_code_error(changeset, message \\ "is invalid or unavailable") do
  Ecto.Changeset.add_error(changeset, :invite_code, message)
end
```

### Anti-Patterns to Avoid

- **Consuming in `Register.valid_invite_code?/1`:** This burns the code before user validation succeeds and violates the requirement that failed registration attempts do not consume invites. [VERIFIED: lib/foglet_bbs/tui/screens/register.ex] [VERIFIED: .planning/phases/03-invite-persistence-and-registration-enforcement/03-SPEC.md]
- **Screen-only invite enforcement:** The TUI must not be the trust boundary; `Foglet.Accounts` must enforce registration and invite state. [VERIFIED: lib/foglet_bbs/authorization.ex]
- **Check-then-update without a transaction/condition:** Loading an invite, checking fields, then updating later can race under concurrent registrations. [CITED: https://www.postgresql.org/docs/16/sql-update.html] [ASSUMED]
- **Casting `issuer_id` or `consumed_by_user_id` from external params:** Project rules say programmatically-set user IDs must not be cast. [VERIFIED: CLAUDE.md]
- **Adding invite status as a persisted enum:** The spec says status is derived from persisted timestamp state. [VERIFIED: .planning/phases/03-invite-persistence-and-registration-enforcement/03-SPEC.md]

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Transactions across user insert and invite consume | Custom rollback flags or manual compensation | `Repo.transaction` / `Ecto.Multi` | Ecto already supports named transactional steps and rollback on step failure. [CITED: https://hexdocs.pm/ecto/Ecto.Multi.html] |
| Unique invite-code enforcement | In-memory set or retry-only uniqueness | DB unique index plus `unique_constraint/3` | Ecto docs rely on database unique constraints and convert violations into changeset errors. [CITED: https://hexdocs.pm/ecto/Ecto.Changeset.html] |
| Authorization | Role checks scattered through invite functions | `Bodyguard.permit(Foglet.Authorization, action, actor, :site)` | Existing policy already defines `:generate_invite` and `:revoke_invite`. [VERIFIED: lib/foglet_bbs/authorization.ex] |
| Runtime invite policy | Hard-coded config values | `Foglet.Config.invite_code_generators/0` and new/existing typed cap accessor | Config is the project runtime source for editable policy. [VERIFIED: lib/foglet_bbs/config.ex] [VERIFIED: docs/ARCHITECTURE.md] |
| Timestamp handling | Custom date/time helpers or deps | `DateTime.utc_now() |> DateTime.truncate(:microsecond)` | Project schemas use `:utc_datetime_usec`, and project rules prefer stdlib date/time. [VERIFIED: docs/DATA_MODEL.md] [VERIFIED: CLAUDE.md] |
| Invite code randomness | Predictable counters or `:rand` | `:crypto.strong_rand_bytes/1` with encoding | `:crypto` is already in the app; generated public codes need unpredictability. [VERIFIED: mix.exs] [ASSUMED] |

**Key insight:** The single-use invariant belongs in Postgres and the Accounts transaction, not in UI flow or BEAM process memory. [VERIFIED: docs/ARCHITECTURE.md] [CITED: https://www.postgresql.org/docs/16/sql-update.html]

## Common Pitfalls

### Pitfall 1: Burning Invites During Preflight

**What goes wrong:** The current registration screen checks `function_exported?(Foglet.Accounts, :consume_invite_code, 1)` and would consume at the invite-code step if that API is added. [VERIFIED: lib/foglet_bbs/tui/screens/register.ex]  
**Why it happens:** The placeholder was designed before invite persistence existed. [VERIFIED: lib/foglet_bbs/tui/screens/register.ex]  
**How to avoid:** Replace the preflight with non-consuming availability validation or remove server-side consumption from the screen; only `Accounts.register_user/1` should consume. [VERIFIED: .planning/phases/03-invite-persistence-and-registration-enforcement/03-SPEC.md]  
**Warning signs:** A test where valid invite + invalid email/password leaves `consumed_at` non-nil. [VERIFIED: .planning/phases/03-invite-persistence-and-registration-enforcement/03-SPEC.md]

### Pitfall 2: Existing `registration_changeset/2` Ignores `invite_code`

**What goes wrong:** Passing `invite_code` through current `register_user/1` has no effect because `User.registration_changeset/2` casts only `:handle`, `:email`, and `:password`. [VERIFIED: lib/foglet_bbs/accounts/user.ex]  
**Why it happens:** Programmatically or workflow-only fields are not currently part of the user schema. [VERIFIED: lib/foglet_bbs/accounts/user.ex]  
**How to avoid:** Keep `invite_code` out of persisted user fields, but use it in `Accounts.register_user/1` orchestration and add errors to the user changeset when invalid. [CITED: https://hexdocs.pm/ecto/Ecto.Changeset.html]  
**Warning signs:** Invite-only tests pass user creation without checking invite state. [VERIFIED: .planning/phases/03-invite-persistence-and-registration-enforcement/03-SPEC.md]

### Pitfall 3: Missing Phase 2 Cap Accessor

**What goes wrong:** The spec requires `Foglet.Config.invite_generation_per_user_limit/0`, but the currently-read `Foglet.Config.Schema` shown in this working tree does not include that key yet. [VERIFIED: lib/foglet_bbs/config/schema.ex]  
**Why it happens:** Phase 3 depends on Phase 2; this research was run while Phase 2 plans exist but are not fully executed in the current tree. [VERIFIED: .planning/ROADMAP.md] [VERIFIED: .planning/STATE.md]  
**How to avoid:** Planner should make Phase 3 tasks conditional on Phase 2 completion, or include a Wave 0 verification task that confirms the cap accessor exists before invite generation implementation. [VERIFIED: .planning/ROADMAP.md]  
**Warning signs:** Compile error for `Foglet.Config.invite_generation_per_user_limit/0` or missing schema seed for `"invite_generation_per_user_limit"`. [VERIFIED: lib/foglet_bbs/config.ex] [VERIFIED: lib/foglet_bbs/config/schema.ex]

### Pitfall 4: Transaction Boundary Drift Around Default Board Subscription

**What goes wrong:** Current `register_user/1` inserts a user, then calls `Foglet.Boards.subscribe_to_defaults/1` after commit. [VERIFIED: lib/foglet_bbs/accounts.ex] If invite-only registration moves user insert into a transaction but leaves invite consumption outside it, success/failure semantics split. [ASSUMED]  
**Why it happens:** The existing subscription choice intentionally does not roll back user creation. [VERIFIED: lib/foglet_bbs/accounts.ex]  
**How to avoid:** Keep invite consumption inside the user-creation transaction, and keep default subscription after successful commit unless requirements explicitly change. [VERIFIED: lib/foglet_bbs/accounts.ex] [VERIFIED: .planning/phases/03-invite-persistence-and-registration-enforcement/03-SPEC.md]  
**Warning signs:** User row exists with unconsumed invite after successful invite-only registration, or consumed invite exists with no user. [VERIFIED: .planning/phases/03-invite-persistence-and-registration-enforcement/03-SPEC.md]

### Pitfall 5: Race Tests That Do Not Actually Race

**What goes wrong:** Sequential tests can prove second redemption fails after first commit, but not concurrent double-use resistance. [ASSUMED]  
**Why it happens:** Ecto sandbox and async tests serialize more than production unless explicitly coordinated. [ASSUMED]  
**How to avoid:** Add at least one deterministic test using two tasks and a shared sandbox connection only if local test infrastructure supports it; otherwise use a sequential second-redemption test plus conditional update count as the core invariant. [ASSUMED]  
**Warning signs:** The only concurrency assertion is "call twice sequentially" without verifying the conditional update path. [ASSUMED]

       :ok <- ensure_invite_policy_allows_generation(actor),
       {:ok, invite} <- insert_unique_invite(actor) do
    {:ok, invite}
  end
end

### Transactional Redemption Shape

```elixir
# Source: Ecto.Multi official docs; PostgreSQL UPDATE official docs
Ecto.Multi.new()
|> Ecto.Multi.insert(:user, User.registration_changeset(%User{}, attrs_without_invite))
|> Ecto.Multi.run(:invite, fn repo, %{user: user} ->
  now = DateTime.utc_now() |> DateTime.truncate(:microsecond)

  query =
    from i in Invite,
      where: i.code == ^invite_code,
      where: is_nil(i.consumed_at),
      where: is_nil(i.revoked_at)

  case repo.update_all(query,
         set: [consumed_at: now, consumed_by_user_id: user.id, updated_at: now]
       ) do
    {1, _} -> {:ok, :consumed}
    {0, _} -> {:error, :invalid_invite_code}
  end
end)
|> Repo.transaction()
```

### Known Threat Patterns for Phoenix/Ecto Invite Registration

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Invite-code guessing | Spoofing | Generate high-entropy codes with `:crypto`, use unique index, avoid predictable counters. [VERIFIED: mix.exs] [ASSUMED] |
| Double redemption race | Tampering | Conditional DB update in same transaction as user insert. [CITED: https://www.postgresql.org/docs/16/sql-update.html] [CITED: https://hexdocs.pm/ecto/Ecto.Multi.html] |
| Authorization bypass through hidden UI only | Elevation of privilege | Enforce `Bodyguard.permit/4` inside `Foglet.Accounts` domain functions. [VERIFIED: lib/foglet_bbs/authorization.ex] |
| SQL injection in invite lookup | Tampering | Use Ecto query interpolation; Ecto queries are sent through Repo, not string-concatenated SQL. [CITED: https://hexdocs.pm/ecto/Ecto.Query.html] |
| User enumeration through invite error detail | Information disclosure | Use one generic `invite_code` error for unknown, revoked, and consumed invite codes unless product explicitly wants detail. [ASSUMED] |
