```elixir
defmodule Foglet.Accounts.Invite do
  use Foglet.Schema

  schema "invite_codes" do
    field :code, :string
    field :consumed_at, :utc_datetime_usec
    field :revoked_at, :utc_datetime_usec

    belongs_to :issuer, Foglet.Accounts.User
    belongs_to :consumed_by_user, Foglet.Accounts.User

    timestamps(type: :utc_datetime_usec)
  end

  @spec changeset(t(), map()) :: Ecto.Changeset.t()
  def changeset(invite, attrs)

  @spec status(t()) :: :available | :consumed | :revoked
  def status(invite)
end
```

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Spoofing | `Foglet.Accounts.Invite.code` | mitigate | Tests require server-generated codes and the schema/migration require a unique `code`; Plan 02 must use OTP `:crypto.strong_rand_bytes/1`, never counters or `:rand`. |
| T-03-02 | Tampering | `invite_codes` consumed fields | mitigate | Migration stores consumed state in the database so Plan 03 can use conditional transactional update. |
| T-03-03 | Elevation of privilege | invite lifecycle tests | mitigate | Lifecycle tests require `Foglet.Accounts` generation/revocation functions to return `{:error, :forbidden}` for unauthorized actors, proving enforcement is domain-side. |
| T-03-04 | Information disclosure | registration tests | mitigate | Registration tests require unknown, revoked, and consumed codes to surface one generic `invite_code` changeset error. |
</threat_model>
