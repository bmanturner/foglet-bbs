# Phase 01.1: Shared Modal Form Primitive — Research

**Researched:** 2026-04-23
**Domain:** Raxol TUI widget composition — stateful form container with focus management
**Confidence:** HIGH (all claims verified against codebase and vendored Raxol docs)

---

## Summary

Phase 01.1 builds `Foglet.TUI.Widgets.Modal.Form` — a stateful modal-overlay widget that hosts a caller-declared list of typed input fields. All major design decisions can be made directly from evidence in the codebase: the widget conventions are codified, the input primitives exist, the focus-management pattern is established by `NewThread`, and the test scaffold is fully legible from existing widget tests.

The central discovery is that **focus management in this codebase is hand-rolled in every screen as a `focused: atom()` field in screen state**, not delegated to any Raxol facility. `Raxol.UI.Components.FocusRing` exists in the framework docs but is not used by any foglet_bbs widget or screen — all foglet_bbs code tracks focus by storing a focused-field identifier (atom or integer) and routing keystroke events only to the focused child. `Modal.Form` should follow the same pattern: store `focus_index :: non_neg_integer()` in its state struct and route character events to the focused child widget state.

Raxol does **not** provide a built-in form container or focus ring used in this project. The three native input primitives already in `lib/foglet_bbs/tui/widgets/input/` cover four of the five field types: `TextInput` (`:text`, `:integer` via coercion), `Checkbox` (`:boolean`), `RadioGroup` (`:enum`). The fifth type — `:textarea` — maps to `Raxol.UI.Components.Input.MultiLineInput`, using the pattern established by `Foglet.TUI.Widgets.Compose`.

**Primary recommendation:** Build `Modal.Form` as a pure D-14 stateful widget (no process) that holds a list of per-field sub-states and routes events by `focus_index`. Compose from existing input widgets and `MultiLineInput` directly; hand-roll nothing that already exists.

## Architectural Responsibility Map

| Capability | Primary Tier | Secondary Tier | Rationale |
|------------|-------------|----------------|-----------|
| Field state storage | Widget struct | — | D-14: all state lives in the widget's own struct, no process |
| Focus index tracking | Widget struct | — | NewThread pattern: `focused` field owned by the stateful container |
| Keystroke routing | `handle_event/2` | — | Modal.Form receives all events; dispatches only to focused child |
| Per-field rendering | Existing input widgets | Compose (textarea) | Reuse TextInput/Checkbox/RadioGroup/MultiLineInput — do not reimplement |
| Typed value coercion | `handle_event/2` or `get_value/1` helper | — | Integer parsing happens at read-time; booleans/enums are native |
| Inline error display | `render/2` | — | Stateless render pass: errors map lives in widget state; rendered beneath each field |
| Modal chrome (border/overlay) | `Foglet.TUI.App.render_modal_overlay` | — | App already centers a `box` with `:double` border around modal content; Modal.Form renders the inner body, not the chrome |
| on_submit / on_cancel callbacks | Caller-owned | — | Callbacks stored in struct at `init/1`; invoked from `handle_event/2` |

### Pattern 1: D-14 Stateful Widget Structure (matches TextInput)

**What:** `init/1 + handle_event/2 + render/2` triplet; all state in a struct; no process.
**When to use:** Any widget that must accumulate keystroke state across renders.

```elixir
# Source: lib/foglet_bbs/tui/widgets/input/text_input.ex (verified)
defmodule Foglet.TUI.Widgets.Input.TextInput do
  defstruct [:raxol_state, :validator, :on_submit, last_action: nil]

  def init(opts) when is_list(opts) do
    # ... build struct from opts ...
    %__MODULE__{ ... }
  end

  @spec handle_event(map(), t()) :: {t(), action()}
  def handle_event(event, %__MODULE__{} = st) do
    # ... return {new_state, action_or_nil} ...
  end

  @spec render(t(), keyword()) :: any()
  def render(%__MODULE__{} = state, opts) do
    %Theme{} = theme = Keyword.fetch!(opts, :theme)
    # ... build view tree using Raxol DSL ...
  end
end
```

**Modal.Form follows this exactly**, with `{new_state, action}` from `handle_event/2` where action is one of: `:submitted`, `:cancelled`, `nil`.

### Pattern 2: Focus Management by Index (matches NewThread)

**What:** Store a `focus_index :: non_neg_integer()` in the widget struct. On Tab, increment with modular wrap. On Shift-Tab, decrement with modular wrap. Events for the focused field are dispatched to that field's sub-state; others are ignored.
**When to use:** Any multi-field container widget.

```elixir
# Source: lib/foglet_bbs/tui/screens/new_thread.ex + state.ex (verified pattern)
# NewThread uses :atom for focus (only 2 fields); Modal.Form uses integer index (N fields)

# Focus advance (Tab)
new_index = rem(focus_index + 1, length(fields))

# Focus retreat (Shift-Tab)
new_index = rem(focus_index - 1 + length(fields), length(fields))

# Route event to focused field only
{new_field_state, _action} = dispatch_to_field(event, Enum.at(fields, focus_index))
```

**Important:** Raxol's `FocusRing` component is documented in the widget gallery but is NOT used anywhere in foglet_bbs. Do not introduce it — it requires component IDs and a different lifecycle than D-14 pure structs.

### Pattern 3: Stateless Child Widgets — Checkbox and RadioGroup

**What:** `Checkbox` and `RadioGroup` are D-16 stateless — they receive state on every render call and have no `handle_event/2`. Modal.Form must store the checked boolean / selected_index in each `%FieldState{}` and pass it to the stateless render function.

```elixir
# Source: lib/foglet_bbs/tui/widgets/input/checkbox.ex (verified)
# Checkbox has no init/1 — caller holds the state
Checkbox.render("Enable feature", checked?: field_state.value, theme: theme)

# Source: lib/foglet_bbs/tui/widgets/input/radio_group.ex (verified)
# RadioGroup has no init/1 — caller holds the selected_index
RadioGroup.render(field.choices, field_state.selected_index, theme: theme)
```

For Checkbox, toggle on Space/Enter. For RadioGroup, navigate with Up/Down inside the focused field.

### Pattern 4: Textarea via Compose + MultiLineInput

**What:** `:textarea` fields use `Raxol.UI.Components.Input.MultiLineInput` directly (via `{:ok, state} = MultiLineInput.init(%{...})`), update via `MultiLineInput.update(message, state)`, and render via `Foglet.TUI.Widgets.Compose.render_input/4`.
**Key:** Key translation goes through `Compose.translate_key/1` — it returns `nil` for Tab/Esc (those stay at the Modal.Form level), and returns a MultiLineInput message for arrow keys, backspace, enter, and character input.

```elixir
# Source: lib/foglet_bbs/tui/widgets/compose.ex (verified)
msg = Compose.translate_key(key_event)
if msg do
  new_mli_state = MultiLineInput.update(msg, field_state.mli_state)
  # store new_mli_state back in field_state
end
```

### Pattern 5: `set_errors/2` — Pure State Update

**What:** Caller invokes this after receiving `:submitted` action and running its own validation. Returns a new `Modal.Form` state with the errors map updated; the form re-renders with error strings beneath each field.

```elixir
# Proposed API (consistent with D-14 pure functions):
@spec set_errors(t(), %{atom() => String.t()}) :: t()
def set_errors(%__MODULE__{} = state, errors) when is_map(errors) do
  %{state | errors: errors}
end
```

### Pattern 6: Inline Error Rendering (matches existing error convention)

Errors render as a single `text/2` line beneath the field widget, using `theme.error.fg`. This matches how `NewThread` surfaces its `ss.error` message (line 119 of new_thread.ex).

```elixir
# Source: lib/foglet_bbs/tui/screens/new_thread.ex line 119 (verified pattern)
text(ss.error, fg: theme.error.fg, style: [:bold])
```

For Modal.Form, render per-field: if `errors[field.name]` is set, append a `text/2` with that message in `theme.error.fg` after the field widget.

### Pattern 7: Focused Label Highlighting

Labels of the focused field use `theme.accent.fg` + `[:bold]`; unfocused labels use `theme.primary.fg` + `[]`. This matches NewThread's `title_label_fg` / `title_label_style` pattern.

```elixir
# Source: lib/foglet_bbs/tui/screens/new_thread.ex lines 101-103 (verified)
title_label_fg = if title_focused, do: theme.accent.fg, else: theme.primary.fg
title_label_style = if title_focused, do: [:bold], else: []
```

### Anti-Patterns to Avoid

- **Spawning a process for Modal.Form state:** D-14 is explicit — no process. State lives in a struct; the caller (App or screen) holds it.
- **Using `Raxol.UI.Components.Modal` (the Raxol framework modal):** The project wraps Raxol's modal differently — `Foglet.TUI.App.render_modal_overlay/2` provides the chrome. Modal.Form renders only the inner body (title, fields, key hint) as the existing `Modal` widget does.
- **Calling `Checkbox.init/1` or `RadioGroup.init/1`:** These don't exist — both are D-16 stateless. Store their state (boolean / integer) directly in `%FieldState{}`.
- **Passing Tab/Shift-Tab to child field widgets:** Tab and Escape must be intercepted at the Modal.Form level before dispatching to child widgets. TextInput.handle_event passes `:submitted` on Enter — Modal.Form must not pass Tab to TextInput (TextInput has no Tab handling; it would fall through as nil, which is safe, but intent is clearer if intercepted first).
- **Building a custom TextArea widget:** `Raxol.UI.Components.Input.MultiLineInput` handles `:textarea` — use it exactly as NewThread does; the key translation contract is already codified in `Compose.translate_key/1`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Single-line text input | Custom char-by-char buffer | `Foglet.TUI.Widgets.Input.TextInput` | Handles cursor, backspace, mask, max_length, Raxol event routing |
| Boolean toggle | Custom render logic | `Foglet.TUI.Widgets.Input.Checkbox` (render only — store boolean in FieldState) | Theme-routed, tested, consistent markers |
| Enum single-select | Custom radio logic | `Foglet.TUI.Widgets.Input.RadioGroup` (render only — store selected_index in FieldState) | Theme-routed, clamp_index, tested |
| Multi-line text area | Custom line buffer + cursor | `Raxol.UI.Components.Input.MultiLineInput` + `Compose.translate_key/1` | Handles undo/redo, cursor_pos, word wrap |
| Key translation for textarea | Per-widget key parsing | `Foglet.TUI.Widgets.Compose.translate_key/1` | Already translates the full keyboard surface; returns nil for Tab/Esc so they stay at form level |
| Modal chrome (border + centering) | box + column per widget | `Foglet.TUI.App.render_modal_overlay/2` | App already centers and borders modal content; Modal.Form renders inner body only |
| Theme color decisions | Hardcoded atoms | `Foglet.TUI.Theme` slots | D-07/D-09 strictly enforced; catalog_smoke_test verifies no leakage |

**Key insight:** Every input primitive this form needs already exists and is tested. The only novel work is the container itself — focus routing, coercion, and the errors API.

## Common Pitfalls

### Pitfall 1: Passing Tab to TextInput

**What goes wrong:** TextInput.handle_event for Tab returns `{state, nil}` (no match → nil action). Focus appears to not advance because the form never updates `focus_index`.
**Why it happens:** Tab is consumed by Modal.Form for focus navigation; if it falls through to child dispatch first, the child swallows it silently.
**How to avoid:** Pattern-match `:tab` and `:tab` with `shift: true` at the TOP of `handle_event/2`, before the child-dispatch path.
**Warning signs:** Tab navigation test passes on field 0→1 but hangs; field state shows no change.

### Pitfall 2: Checkbox and RadioGroup Have No `init/1`

**What goes wrong:** Calling `Checkbox.init(...)` or `RadioGroup.init(...)` raises `UndefinedFunctionError` — both are D-16 stateless and expose only `render/2` (and `render/3` for RadioGroup).
**Why it happens:** Stateless widgets have no state struct — the caller stores the value.
**How to avoid:** `FieldState` for `:boolean` holds a raw boolean; for `:enum` holds a raw `selected_index` integer. Pass these directly to the render functions.

### Pitfall 3: Integer Coercion — Empty Field

**What goes wrong:** `String.to_integer("")` raises `ArgumentError`. On submit, an empty integer field crashes.
**Why it happens:** User may submit without typing anything in an integer field.
**How to avoid:** Use `Integer.parse/1` which returns `:error` on failure, then map `:error → nil`. The spec says integer fields deliver `integer() | nil`; `nil` tells the caller the field was blank/invalid.

```elixir
# Correct coercion
case Integer.parse(String.trim(text_value)) do
  {n, ""} -> n
  _ -> nil
end
```

### Pitfall 4: Modal.Form Renders Its Own Chrome

**What goes wrong:** Modal.Form renders its own `box border: :double` wrapper and then `render_modal_overlay` wraps it in another border — double-bordered box.
**Why it happens:** Existing `Modal.render/2` renders inner body only; the border lives in `render_modal_overlay`. Modal.Form should match.
**How to avoid:** `render/2` returns a bare `column` (title, divider, fields, key hint) — no outer box. The app positions it.

### Pitfall 5: Shift-Tab Event Shape

**What goes wrong:** Shift-Tab tested as `%{key: :tab}` — test passes but Shift-Tab is not wired at all.
**Why it happens:** Shift-Tab arrives as `%{key: :tab, shift: true}` or `%{key: :tab, modifiers: [:shift]}` — the exact shape depends on Raxol's InputParser. The codebase does not have an existing Shift-Tab handler to copy from.
**How to avoid:** Check the actual event shape. Grep for how `ctrl: true` is used (seen in `handle_compose_key` in new_thread.ex at line 237) — same pattern: `%{key: :char, char: "c", ctrl: true}`. For Shift-Tab, it is likely `%{key: :tab, shift: true}`. Verify by checking `Raxol.Core.Events.InputParser` or the InputParser tests in vendor/raxol.
**Warning signs:** Shift-Tab test fails even though Tab passes.

### Pitfall 6: RadioGroup `render/3` vs `render/2`

**What goes wrong:** Calling `RadioGroup.render(options, selected_index, theme: theme)` — but the arity is 3, not 2. The call is correct, but if you accidentally call `RadioGroup.render(options, selected_index)` (missing opts), it crashes with FunctionClauseError.
**How to avoid:** Always pass `theme: theme` as the third argument keyword list. RadioGroup.render/3 calls `Keyword.fetch!(:theme)`.

### Field Dispatch Pattern

```elixir
# Source pattern: lib/foglet_bbs/tui/screens/new_thread.ex (verified focus routing)
def handle_event(%{key: :escape}, %__MODULE__{on_cancel: cb} = state) do
  cb.()
  {state, :cancelled}
end

def handle_event(%{key: :tab} = _event, state) do
  n = length(state.fields)
  new_idx = rem(state.focus_index + 1, n)
  {%{state | focus_index: new_idx}, nil}
end

def handle_event(%{key: :tab, shift: true} = _event, state) do
  n = length(state.fields)
  new_idx = rem(state.focus_index - 1 + n, n)
  {%{state | focus_index: new_idx}, nil}
end

def handle_event(%{key: :enter}, state) do
  last_idx = length(state.fields) - 1
  if state.focus_index == last_idx do
    payload = collect_values(state)
    state.on_submit.(payload)
    {state, :submitted}
  else
    # Advance focus like Tab
    handle_event(%{key: :tab}, state)
  end
end

def handle_event(event, state) do
  # Dispatch to focused field
  focused_spec = Enum.at(state.fields, state.focus_index)
  focused_st = Enum.at(state.field_states, state.focus_index)
  {new_field_st, _} = dispatch_to_field(event, focused_spec, focused_st)
  new_field_states = List.replace_at(state.field_states, state.focus_index, new_field_st)
  {%{state | field_states: new_field_states}, nil}
end
```

### Render Pattern (from Modal.render/2 + NewThread pattern)

```elixir
# Source pattern: lib/foglet_bbs/tui/widgets/modal.ex + new_thread.ex (verified)
def render(%__MODULE__{} = state, opts) do
  %Theme{} = theme = Keyword.fetch!(opts, :theme)

  field_rows =
    state.fields
    |> Enum.with_index()
    |> Enum.flat_map(fn {spec, idx} ->
      focused? = idx == state.focus_index
      field_st = Enum.at(state.field_states, idx)
      label_fg = if focused?, do: theme.accent.fg, else: theme.primary.fg
      label_style = if focused?, do: [:bold], else: []
      error_msg = Map.get(state.errors, spec.name)

      field_widget = render_field(spec, field_st, focused?, theme)
      error_row = if error_msg, do: [text(error_msg, fg: theme.error.fg)], else: []

      [text(spec.label, fg: label_fg, style: label_style), field_widget] ++ error_row
    end)

  column [] do
    [text(" #{state.title} ", fg: theme.title.fg, style: [:bold]), divider()] ++
      field_rows ++
      [text("[Tab] Next  [Shift+Tab] Prev  [Enter] Submit  [Esc] Cancel", fg: theme.dim.fg)]
  end
end
```

## The NewThread Anti-Pattern

The spec calls out `NewThread`-style inline sub-step state machines as the pattern NOT to generalize. Here is what that looks like and why it does not scale:

- `NewThread.State` has `step: :board | :compose`, `focused: :title | :body` — hardcoded to exactly 2 field atoms.
- Adding a third field (e.g., tags) requires adding a new focus atom, new pattern match clauses in `handle_compose_key`, new render branch, new `ss.tag_input_state` field.
- No shared focus navigation logic exists — every screen re-implements `focused =` toggle.
- The key hint is hardcoded ("Tab — Switch field" or "Tab — Preview/Edit").

`Modal.Form` generalizes this by: (1) accepting N fields as a list at `init/1`, (2) storing `focus_index :: integer` instead of a named atom, (3) handling Tab/Shift-Tab/Enter/Esc uniformly regardless of field count or type.

Phase 2+ screens will call `Modal.Form.init([fields: [...], ...])` and dispatch all keystrokes to the form — zero per-screen focus machinery.
