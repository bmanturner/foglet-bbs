# Phase 01.1: Shared Modal Form Primitive — Specification

**Created:** 2026-04-23
**Ambiguity score:** 0.15 (gate: ≤ 0.20)
**Requirements:** 9 locked

## Goal

Deliver `Foglet.TUI.Widgets.Modal.Form` — a stateful modal-overlay widget that hosts a caller-declared list of typed input fields (text, integer, boolean, enum, multiline textarea), supports wrap-around Tab/Shift-Tab focus navigation, submits via `on_submit/1` with a `%{field_name => value}` map, cancels via `on_cancel/0` on Esc, and renders caller-supplied per-field inline errors without dismissing — so Phase 2+ CRUD workflows share one container instead of each screen re-inventing a per-step machine.

## Background

`Foglet.TUI.Widgets.Modal` (`lib/foglet_bbs/tui/widgets/modal.ex`) today is a stateless body renderer supporting only `info | error | warning | confirm` variants — it has no input-field concept. Input primitives already exist under `lib/foglet_bbs/tui/widgets/input/` (`TextInput`, `Checkbox`, `RadioGroup`, `Button`) and all follow the D-14 `init/1 + handle_event/2 + render/2` triplet with D-13 `theme:` kwarg routing. `NewThread`-style inline sub-step state machines in `lib/foglet_bbs/tui/screens/new_thread/` do not generalize across the upcoming Phase 2 board/category CRUD flows (INVT-06/07, SYSO-02/03/04). No typed form-container widget exists in the codebase.

## Requirements

1. **Modal.Form module exists**: A new module `Foglet.TUI.Widgets.Modal.Form` is created following D-14 widget conventions.
   - Current: No form-container widget exists; callers compose input widgets ad-hoc in screen state.
   - Target: `lib/foglet_bbs/tui/widgets/modal/form.ex` exposes `init/1`, `handle_event/2`, `render/2`; `@moduledoc` cites D-07, D-09, D-13, D-14.
   - Acceptance: Module compiles; D-18 theme-hygiene + smoke-render test passes.

2. **Supports five field types**: Fields declared by the caller can be `:text`, `:integer`, `:boolean`, `:enum`, or `:textarea` (multiline).
   - Current: No typed-field container exists.
   - Target: `init/1` accepts a `:fields` list where each entry specifies `name`, `type` (one of the five), `label`, and type-specific opts (e.g., enum `:choices`, text `:max_length`, textarea `:rows`).
   - Acceptance: Unit test constructs a form with one field of each of the five types and renders without error.

3. **Typed value coercion on submit**: The submit payload delivers values in their native Elixir types.
   - Current: N/A — no coercion layer exists.
   - Target: `:text` → `String.t()`; `:integer` → `integer()` (or `nil` when empty/invalid so the caller's validator can reject); `:boolean` → `boolean()`; `:enum` → one of the declared choice atoms/strings; `:textarea` → `String.t()` (may contain `\n`).
   - Acceptance: Test populates one field of each type and asserts `on_submit` receives a map whose values match the declared types.

4. **Tab / Shift-Tab navigation with wrap**: Focus moves between fields in declaration order and wraps at both ends.
   - Current: No shared focus navigator exists for stacked fields.
   - Target: Tab advances focus; Shift-Tab retreats; Tab from the last field wraps to the first; Shift-Tab from the first wraps to the last. Only the focused field receives keystroke events.
   - Acceptance: Test drives Tab/Shift-Tab through a 3-field form and asserts focus index sequence `0 → 1 → 2 → 0` and `0 → 2 → 1 → 0`.

5. **Enter on last field submits**: Enter pressed while the last field is focused triggers submission.
   - Current: No submit trigger exists.
   - Target: When focus is on the last field and the event is `:enter`, the form invokes `on_submit` with the collected `%{field_name => value}` map. Enter on non-last fields advances focus (equivalent to Tab).
   - Acceptance: Test focuses the last field, sends `:enter`, and asserts the `on_submit` callback fires once with the expected map.

6. **Esc cancels unconditionally**: Esc triggers `on_cancel` with no dirty-state prompt.
   - Current: N/A.
   - Target: On `:escape`, the form invokes `on_cancel` (0-arity); the primitive itself does not prompt or track dirtiness.
   - Acceptance: Test sends `:escape` after mutating a field and asserts `on_cancel` fires once and no `on_submit` callback fires.

7. **Caller-supplied per-field errors render inline without dismissing**: When `on_submit` returns `{:error, %{field_name => message}}`, the modal remains open and renders each message under its field.
   - Current: N/A — no error-render contract exists.
   - Target: The host screen, upon receiving the form's submit action, runs its own validation and calls a form API (e.g., `Modal.Form.set_errors/2`) to update state; the form then renders each `message` string beneath the corresponding field in the theme's `error` slot color. `on_submit` returning `:ok` closes/dismisses the modal (caller-orchestrated).
   - Acceptance: Test sets errors for two of three fields, re-renders, and asserts both messages appear in the rendered view tree under the matching field labels, and that the modal is still present.

8. **Existing Modal unchanged**: The info/warning/error/confirm `Modal` widget continues to function identically.
   - Current: `Foglet.TUI.Widgets.Modal` handles 4 variants and is used by existing screens.
   - Target: No edits to `modal.ex`; `Modal.Form` lives in a sibling file (`modal/form.ex`).
   - Acceptance: Existing `test/foglet_bbs/tui/widgets/modal_test.exs` (if present) continues to pass unchanged; grep confirms no diff to `modal.ex`.

9. **End-to-end fixture test**: A test exercises open → navigate → submit.
   - Current: No form-level integration test exists.
   - Target: Test in `test/foglet_bbs/tui/widgets/modal/form_test.exs` initializes a form with at least one field of each supported type, drives keystrokes through Tab navigation, populates each field, submits with Enter, and asserts the callback payload.
   - Acceptance: Test file exists and passes under `mix test`.

## Boundaries

**In scope:**
- New module `Foglet.TUI.Widgets.Modal.Form` with `init/1`, `handle_event/2`, `render/2`.
- Field types: `:text`, `:integer`, `:boolean`, `:enum`, `:textarea`.
- Wrap-around Tab/Shift-Tab focus navigation; Enter-on-last submits; Esc cancels.
- Caller-supplied inline per-field errors set via a `set_errors/2` (or equivalent) API; rendered in theme error slot.
- `on_submit/1` delivers a `%{field_name => typed_value}` map; `on_cancel/0` fires on Esc.
- D-18 widget test: theme hygiene + smoke render + end-to-end fixture (open → navigate → submit).
- Update `lib/foglet_bbs/tui/widgets/README.md` with a Modal.Form entry.

**Out of scope:**
- Password/masked fields — `TextInput` already supports `:mask_char`; add when a CRUD flow asks for it.
- Date/time fields — no caller needs them yet.
- Built-in validation (regex, required, length) — validation stays caller-owned via `on_submit`-returned errors; primitive is a dumb container.
- Dirty-state cancel prompt — primitive cancels unconditionally; wrapping with a confirm is the caller's concern.
- Rewiring existing screens (`NewThread`, etc.) onto Modal.Form — this phase delivers the primitive only; adoption happens in Phase 2+ as flows land.
- Multi-column / grouped / tabbed form layouts — single vertical stack only.
- Async/remote validation — all validation is synchronous caller code.

## Constraints

- Must follow D-07/D-09 theme routing — no hardcoded color atoms.
- Must accept `theme:` as a render kwarg (D-13).
- Must follow D-14 stateful-widget triplet (`init/1 + handle_event/2 + render/2`); no spawned process.
- File location `lib/foglet_bbs/tui/widgets/modal/form.ex` (D-10 bucket layout — `modal/` sibling of `modal.ex` is acceptable since `Modal` itself is flat; one module per file, per project CLAUDE.md).
- Must pass `mix precommit` (format, credo --strict, sobelow, dialyzer, warnings-as-errors).
- Reuse existing input primitives (`TextInput`, `Checkbox`, `RadioGroup`) rather than reimplementing field UI — except where textarea/enum need a new internal adapter.
