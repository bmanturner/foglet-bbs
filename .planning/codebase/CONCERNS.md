## Tech Debt

### Invite Code Validation is a Stub

**Files:** `lib/foglet_bbs/tui/screens/register.ex` (lines 418–432)

When `registration_mode` is `"invite_only"`, the TUI collects and "validates" an invite code, but there is no `invite_codes` table in the DB and no `Foglet.Accounts.consume_invite_code/1` function. The code path uses `function_exported?/2` at runtime to detect the missing function and falls back to accepting any non-empty alphanumeric string (4–32 chars) as a valid code. Selecting `"invite_only"` mode in production would silently accept any string as a valid code.

```elixir
# register.ex — the fallback path
defp valid_invite_code?(code) when is_binary(code) and byte_size(code) > 0 do
  if function_exported?(Foglet.Accounts, :consume_invite_code, 1) do
    ...
  else
    # Accept any non-empty alphanumeric code when invite_codes table isn't ready.
    Regex.match?(~r/\A[A-Za-z0-9]{4,32}\z/, code)
  end
end
```

The moduledoc explicitly notes this is tracked for Phase 8 (`D-04, D-05`). No migration for `invite_codes` exists.

**Impact:** `registration_mode = "invite_only"` is non-functional in prod. Enabling it would make invite enforcement illusory.

### Email Verification Has No Email Delivery

**Files:** `lib/foglet_bbs/accounts.ex` (lines 299–330), `lib/foglet_bbs/tui/screens/register.ex`, `lib/foglet_bbs/tui/screens/login.ex`

`Foglet.Accounts.deliver_user_confirmation_instructions/2` and `deliver_user_reset_password_instructions/2` generate and persist tokens but never send email. Phase 10 is noted as the target for Swoosh delivery. In the interim, verify codes are only surfaced via a `Logger.info` call gated by `Application.compile_env(:foglet_bbs, :log_verify_codes, false)` — enabled in `dev.exs`, never in prod.

**Impact:** In production with `require_email_verification: true`, new users reach the verify screen but cannot receive their code. They are permanently stuck unless a sysop reads the server logs. Password reset is similarly log-only.

### `Accounts.delete_user/1` Leaves Posts Orphaned

**Files:** `lib/foglet_bbs/accounts.ex` (lines 241–260)

The docstring acknowledges that post authorship rewriting (`posts.user_id → tombstone_user_id`) is deferred to "Phase 2+". Currently `delete_user/1` clears PII on the user row but does not rewrite any posts. Deleted user's posts retain their original `user_id` FK, which references a soft-deleted row. The `user_id` FK is `ON DELETE SET NULL` or similar at the DB level, but the semantic tombstoning intended by the roadmap is absent.

**Impact:** Posts from deleted users display a dangling or nil author in the TUI. No crash — `PostCard` handles nil `:user` gracefully — but the intended UX (show `[deleted]` attribution) is not implemented.

### Dual Domain-Injection Patterns in the TUI Layer

**Files:** `lib/foglet_bbs/tui/app.ex` (lines 365–498), `lib/foglet_bbs/tui/screens/domain.ex`

Screens use `Foglet.TUI.Screens.Domain.get/2` to inject domain modules (testable, explicit). `TUI.App.do_update/2` closures use raw `get_in(ctx, [:domain, :boards]) || Foglet.Boards` instead. The comment at line 365 explicitly calls this out as intentional but tracked for migration:

> "Intentionally not using Domain.get/2 — do_update closures snapshot the domain module directly so the task closure captures only the atom, not the full state map. Tracked for migration in a future phase."

Two competing idioms for the same concept in the same code layer is a source of confusion for new contributors.

**Impact:** Low — functional parity. Anyone adding a new `do_update` I/O clause will need to know which pattern to follow and why.

### `Threads.list_threads/2` Returns Maps Not Structs

**Files:** `lib/foglet_bbs/threads.ex` (lines 72–124)

`list_threads/2` (the 2-arity user-aware version) returns plain maps with `has_unread: boolean` merged in. `list_threads/1` returns `[Thread.t()]` structs. Callers must be aware of this type divergence. `ThreadList` and `TUI.App` both call the 2-arity form but the `@spec` only loosely documents this:

```elixir
@spec list_threads(String.t(), String.t() | nil) :: [map()]
```

The manual `preload_created_by/1` implementation reimplements Ecto's association preloading by issuing a second query and mapping over results in memory. For very large boards this could be slow.

### SSH Host Key Directory in Production Is Not Validated

**Files:** `config/runtime.exs` (line 78), `lib/foglet_bbs/ssh/daemon_owner.ex`

`runtime.exs` reads `SSH_HOST_KEY_DIR` from env (defaulting to `"priv/ssh"`), but there is no startup check that the directory exists or contains valid host keys. A missing directory causes the SSH daemon to fail at the `:ssh.daemon/2` call with an opaque Erlang error rather than a clear diagnostic.

**Impact:** Silent deploy failure — the BBS appears to start but accepts no SSH connections. `mix foglet.doctor` exists but its SSH key check may not cover this case.

## Known Weak Spots / Fragile Areas

### `TUI.App.do_update/2` Is Becoming a God Function

**Files:** `lib/foglet_bbs/tui/app.ex` (792 lines)

`do_update/2` has ~30 clauses handling navigation, I/O dispatch, PubSub messages, session management, modal callbacks, and heartbeats. While each clause is individually small, the aggregate is large. Adding new screens or features requires editing this central routing table.

The `process_screen_commands/2` helper (line 703) also contains a general-purpose `do_update/2` re-dispatch loop that makes it hard to statically reason about command execution order.

**Safe modification:** Add new `do_update/2` clauses following the existing `{:load_X}` / `{:X_loaded, result}` task pair pattern. The existing structure is consistent and the catch-all clause prevents hard crashes on unknown messages.

### `handle_key/2` Clause Order Is Load-Bearing in Three Screens

**Files:**
- `lib/foglet_bbs/tui/screens/new_thread.ex` (lines 281–282)
- `lib/foglet_bbs/tui/screens/post_composer.ex` (line 90)

Both screens have explicit `# NOTE: source order matters for handle_key/2 clauses — do not reorder` comments. The Ctrl+C / Ctrl+S intercept clauses rely on being matched before more general character clauses. A routine Credo-driven reorder or merge would silently break keyboard shortcuts.

**Risk:** Someone unfamiliar with this constraint reformats the file and reorders clauses. No test directly asserts on clause order.

---

### `Foglet.SSH.PubkeyStash` Has No TTL Eviction

**Files:** `lib/foglet_bbs/ssh/pubkey_stash.ex` (lines 22–27)

The module doc acknowledges this: "A low-priority periodic sweep is not implemented — the stash is bounded by max_sessions (500) and each entry is a small Erlang term." Entries survive only if a connection completes the `ssh_channel_up` callback. If a connection drops after `is_auth_key/3` but before `handle_msg({:ssh_channel_up, ...}, ...)`, the entry leaks indefinitely.

**Impact:** Very low at current scale — bounded by `@max_connections = 500`. Would be a concern under connection-flood attacks.

### `CLIHandler.get_dispatcher/1` Uses `GenServer.call` + `Process.alive?` Anti-pattern

**Files:** `lib/foglet_bbs/ssh/cli_handler.ex` (lines 361–370)

```elixir
defp get_dispatcher(lifecycle_pid) do
  if Process.alive?(lifecycle_pid) do
    %{dispatcher_pid: pid} = GenServer.call(lifecycle_pid, :get_full_state)
    pid
  else
    nil
  end
catch
  :exit, _ -> nil
end
```

`Process.alive?/1` has a TOCTOU race: the process can die between the check and the `GenServer.call`. The `catch :exit, _` clause covers this, making the alive check redundant (though harmless). The CLAUDE.md guidelines flag `Process.alive?/1` as an anti-pattern to avoid in production code.

## Test Coverage Gaps

### SSH Channel Lifecycle Has No Integration Tests

**Files:** `lib/foglet_bbs/ssh/cli_handler.ex` (434 lines), `test/foglet_bbs/ssh/cli_handler_test.exs`

`cli_handler_test.exs` tests pubkey stash correlation and context-shape only. The test module's own `@moduledoc` states:

> "Full SSH channel event handling requires a real SSH harness (D-21 — out of scope). These tests exercise the pubkey correlation and context-building paths..."

This means none of the following are tested: `handle_ssh_msg` for `:pty`, `:data`, `:window_change`, `:eof`, `:closed`; the alt-screen LEAVE escape sequence; the CRLF normalization in `make_crlf_writer`; or the `handle_msg({:EXIT, ...})` lifecycle crash handler.

**Risk:** Regressions in disconnect handling, terminal rendering artifacts, or connection limit enforcement go undetected.

**Priority:** Medium — this is the outermost layer and the hardest to test; integration tests would require spawning a real SSH client.

### `supervisor_test.exs` Uses `Process.alive?` for Assertions (CLAUDE.md Anti-pattern)

**Files:** `test/foglet_bbs/sessions/supervisor_test.exs`

The test file uses `Process.alive?(pid)` for assertions in multiple tests (lines 22, 40, 76, 92, 93). CLAUDE.md explicitly warns against this:

> "Avoid `Process.alive?/1`. Synchronize deterministically instead."

The correct pattern is monitoring with `assert_receive {:DOWN, ...}` for death detection, and `:sys.get_state/1` for liveness. The supervisor tests partially follow the correct pattern (several tests use `assert_receive {:DOWN, ...}`) but are inconsistent.

**Impact:** Tests may pass or fail based on scheduler timing, especially under heavy load or in CI.

### PubSub Broadcasting Is Wired but Never Emitted

**Files:** `lib/foglet_bbs/tui/app.ex` (lines 211–244), `lib/foglet_bbs/tui/pub_sub_forwarder.ex`, `lib/foglet_bbs/boards/server.ex`

`TUI.App` subscribes to `"boards"`, `"board:<id>"`, `"thread:<id>"`, and `"user:<id>"` topics via `PubSubForwarder`. However, no production code in `lib/` calls `Phoenix.PubSub.broadcast/3` — only `Phoenix.PubSub.subscribe/2` is called. The real-time update handlers (`{:board_activity, ...}`, `{:thread_activity, ...}`, `{:notification, ...}`) in `TUI.App.do_update/2` are therefore dead code in the current state.

**Impact:** Users see no live updates when another user posts. They must navigate away and back. Tests that exercise PubSub handlers would need to manually broadcast.

**Priority:** High for multi-user experience — the BBS feels static without it.

### Soft-Deleted Posts Not Rendered as Tombstones in the TUI

**Files:** `lib/foglet_bbs/posts.ex` (line 90), `lib/foglet_bbs/tui/widgets/post/post_card.ex`

`Posts.list_posts/1` filters out deleted posts entirely (`where: is_nil(p.deleted_at)`). The `Posts.delete_post/1` docstring says deleted posts "should be shown as a tombstone (`[deleted]`)," but the query excludes them. BBS convention typically shows tombstones to preserve thread numbering context.

**Impact:** Post numbering in a thread appears to skip silently when posts are deleted. A user reading "Post 1 of 2" does not know if post 2 was originally post 3.

### No Role-Based Authorization in the TUI

**Files:** `lib/foglet_bbs/tui/screens/`, `lib/foglet_bbs/accounts/user.ex`

`User` has `:role` (`:user`, `:mod`, `:sysop`) and `:status` (`:active`, `:pending`, `:suspended`). The `login.ex` screen blocks `:pending` and `:suspended` users. However, there is no authorization check anywhere in the TUI that gates mod/sysop operations (lock thread, sticky thread, move thread, delete post) behind a role check. The context functions themselves (`Threads.lock_thread/1`, `Posts.delete_post/2`) have no authorization layer — they operate on whoever calls them.

**Impact:** Currently not exploitable because there are no TUI screens that expose these operations to regular users — but adding a moderation screen without adding role checks would be an easy mistake to make.

## Performance Concerns

### `Boards.unread_counts/1` Issues a 3-Way JOIN on Every Board List Load

**Files:** `lib/foglet_bbs/boards.ex` (lines 275–290)

Every call to `list_subscribed_boards/1` triggers `unread_counts/1`, which issues a query with `Subscription LEFT JOIN ReadPointer LEFT JOIN Post` grouped by `board_id`. This is a non-trivial query that touches the `posts` table for every board subscription. At small scale (personal BBS) this is fine, but at moderate scale (thousands of posts across many boards) it may become slow.

No caching exists for unread counts. Every board list render (including PubSub-triggered reloads) re-runs this query.

---

### `Markdown.render/1` Called Per-Frame Without Full Caching

**Files:** `lib/foglet_bbs/markdown.ex`, `lib/foglet_bbs/tui/screens/post_reader.ex`

`PostReader` caches rendered markdown in `render_cache` keyed by `{post_id, width}`. However, the cache is discarded on `Q` (screen exit) and rebuilt on re-entry. In a thread with many posts, re-entering the thread re-parses all previously-rendered posts. `Foglet.Markdown.render/1` calls `MDEx.to_html!/1` which is an NIF call, making repeated parses on the Lifecycle process potentially slow for post-heavy threads.

---

### `Threads.preload_created_by/1` Is a Manual N+1 Avoidance With In-Memory Assembly

**Files:** `lib/foglet_bbs/threads.ex` (lines 107–124)

`list_threads/2` returns plain maps and cannot use `Repo.preload/2` (which only works on Ecto structs). The custom `preload_created_by/1` assembles the users map in memory with `Map.new/2` and calls `Repo.all` once. This is correct and efficient, but it diverges from the standard Ecto preload pattern and could confuse contributors who expect `thread.created_by` to be a preloaded Ecto association.

## Security Considerations

### Database SSL Disabled by Default in Production Config

**Files:** `config/runtime.exs` (line 42)

```elixir
# ssl: true,
```

The production database configuration has SSL commented out. For a deployment where the DB is not co-located with the app on the same host, traffic between app and database would be unencrypted.

**Impact:** Medium — depends on deployment topology. Add `ssl: true` and configure certificates before any non-local deployment.

---

### `no_auth_needed: true` SSH Daemon Requires Security Note Awareness

**Files:** `lib/foglet_bbs/ssh/key_cb.ex`, `lib/foglet_bbs/ssh/daemon_owner.ex`

The SSH daemon is configured with `no_auth_needed: true` (all connections accepted at the SSH protocol layer). Identity resolution happens inside the TUI via `PubkeyStash` correlation. This is intentional and documented (D-13), but means:

1. Any client can connect regardless of whether they have a registered key.
2. Key-less connections become guests and see the login/register screens.
3. There is no SSH-layer rate limiting on connection attempts — only the `@max_connections = 500` ETS counter.

A brute-force registration or login attempt via scripted SSH connections is not currently mitigated at the SSH layer. The `@max_connections` cap is the only guard.

---

### Dev Secret Keys Committed to Version Control

**Files:** `config/dev.exs` (line 26), `config/test.exs` (line 20)

Static `secret_key_base` values are committed for `dev` and `test` environments. This is normal Phoenix practice for local dev. Production correctly reads `SECRET_KEY_BASE` from the environment (`config/runtime.exs`). No concern for prod, but new contributors should be told not to use these values outside local dev.

## Things That Would Surprise a New Contributor

1. **`TUI.App` owns all I/O dispatch** — screens return symbolic command tuples (`{:load_boards}`, `{:load_threads, id}`) rather than Ecto calls. All DB work is routed through `TUI.App.do_update/2` closures into `Command.task` off-process tasks. A screen that calls `Foglet.Boards.list_boards/0` directly would block the Lifecycle process.

2. **`register.ex` has two `init_screen_state` paths** — `init_screen_state/1` always returns `mode: "open"` regardless of config. The runtime-correct initial state comes from lazy `get_register_ss/1` on first render, which calls `init_screen_state_for/1` with the actual `session_context`. The `@doc` for `init_screen_state/1` documents this divergence.

3. **`handle_key/2` clause order is load-bearing** in `new_thread.ex` and `post_composer.ex`. Do not let Credo or a formatter reorder these clauses.

4. **Oban is in deps but never used** — `mix deps.get` and dialyzer PLT builds include Oban for zero current benefit.

5. **PubSub subscriptions are wired but no broadcasts are emitted** — the real-time update infrastructure is half-built. Adding a `Phoenix.PubSub.broadcast/3` call in `Boards.Server` (or elsewhere) would immediately activate the live-update handlers in `TUI.App`.

6. **Dialyzer has a known-noise ignore list** in `.dialyzer_ignore.exs` covering 20+ files with `:contract_supertype` and `:unknown_type` warnings from Ecto schema `t/0` types and Raxol widget render specs. These are false positives, not real bugs. When touching these files, check whether the warning can now be fixed and the ignore entry removed.

7. **`Process.alive?/1` appears in both `lib/` and `test/`** — `CLIHandler` uses it with a `catch :exit, _` safety net (making it harmless but redundant); `supervisor_test.exs` uses it for liveness assertions (contra CLAUDE.md guidance). Neither causes bugs today but both create timing-sensitive surface.
