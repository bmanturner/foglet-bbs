## Pattern Overview

**Overall:** Single OTP application (`:foglet_bbs`) on the BEAM with two independent network interfaces — an SSH server driving a terminal TUI and a Phoenix endpoint serving LiveDashboard. Both interfaces converge on the same session and domain layers.

**Key Characteristics:**
- SSH-first: the primary user interface is a TUI over SSH; there is intentionally no web UI for end users.
- Interface-agnostic Session layer: one GenServer per authenticated user, enforcing a one-session-per-user rule across interfaces.
- Domain contexts own all persistence (Ecto + Postgres); domain state is never duplicated into the TUI.
- Raxol (vendored at `vendor/raxol`) drives the terminal rendering lifecycle — the TUI App module implements `Raxol.Core.Runtime.Application` and is started by `Raxol.Core.Runtime.Lifecycle`.
- ETS tables carry all ephemeral state (config cache, pubkey stash, connection counter); nothing in ETS is required for correctness after a restart.

## Layers

**SSH Interface Layer:**
- Purpose: Accept incoming SSH connections, authenticate users, start TUI lifecycle per connection.
- Location: `lib/foglet_bbs/ssh/`
- Contains: `CLIHandler` (`:ssh_server_channel` behaviour), `DaemonOwner`, `KeyCB` (Erlang `:ssh` key callback), `PubkeyStash` (ETS), `Supervisor`.
- Depends on: Erlang `:ssh` application, `Foglet.Sessions.Supervisor`, `Foglet.Accounts`, `Raxol.SSH.IOAdapter`, `Raxol.Core.Runtime.Lifecycle`.
- Used by: Erlang `:ssh` daemon (calls into `CLIHandler` via `:ssh_server_channel` callbacks).

**TUI Layer:**
- Purpose: Render the BBS user interface and route user input. Runs inside a Raxol Lifecycle process per connection.
- Location: `lib/foglet_bbs/tui/`
- Contains: `App` (top-level Raxol application), `Screen` behaviour, `screens/*` (one module per screen), `widgets/*` (reusable Raxol view components), `Theme`, `PubSubForwarder`, `SizeGate`, `Modal`, `Command`, `Input`.
- Depends on: Domain contexts (`Foglet.Boards`, `Foglet.Threads`, `Foglet.Posts`, `Foglet.Accounts`, `Foglet.Markdown`), `Foglet.Sessions.Session`, `Foglet.Config`, Phoenix.PubSub.
- Used by: `Foglet.SSH.CLIHandler` (starts the Lifecycle), `Foglet.Sessions.Supervisor` (for session promotion).

**Session Layer:**
- Purpose: Long-lived per-user process that holds session identity, terminal size, and TUI pid. Enforces one-session-per-user.
- Location: `lib/foglet_bbs/sessions/`
- Contains: `Session` (GenServer), `Supervisor` (DynamicSupervisor with replacement protocol).
- Depends on: `Foglet.Sessions.Registry` (local Registry with `:unique` keys keyed by `user_id`).
- Used by: `CLIHandler`, `Foglet.TUI.App`, `Foglet.SSH.PubkeyStash`, `Foglet.Boards`.

**Domain Core:**
- Purpose: Business logic and persistence for all BBS entities.
- Location: `lib/foglet_bbs/` (context modules at the top level: `accounts.ex`, `boards.ex`, `threads.ex`, `posts.ex`, `config.ex`, `markdown.ex`)
- Contains: Context modules (public APIs), Ecto schema modules in subdirectories (e.g. `accounts/user.ex`, `boards/board.ex`, `posts/post.ex`).
- Depends on: `FogletBbs.Repo`, `Foglet.Schema` (shared UUID/timestamp defaults).
- Used by: TUI layer, SSH layer, Mix tasks.

**Phoenix Web Layer:**
- Purpose: LiveDashboard observability for sysops. No end-user web UI.
- Location: `lib/foglet_bbs_web/`
- Contains: `Endpoint`, `Router`, `Telemetry`, `FogletBbsWeb` (macro helpers), `controllers/error_json.ex`.
- Depends on: Phoenix, Phoenix LiveDashboard, Bandit (HTTP adapter).
- Used by: Sysop browser connections only.

**Board Server Layer:**
- Purpose: One GenServer per active board, serializing message-number allocation.
- Location: `lib/foglet_bbs/boards/server.ex`, `lib/foglet_bbs/boards/supervisor.ex`
- Contains: `Boards.Server` (GenServer), `Boards.Supervisor` (DynamicSupervisor).
- Depends on: `Foglet.BoardRegistry` (local Registry), `FogletBbs.Repo`, `Ecto.Multi`.
- Used by: `Foglet.Posts` context (calls `Boards.Server.create_post/4` and `create_thread/3`).

## Data Flow

**SSH Connection → TUI Render:**

1. Erlang `:ssh` daemon accepts connection → calls `Foglet.SSH.KeyCB.is_auth_key/3`, which stashes the offered public key in `Foglet.SSH.PubkeyStash` (ETS).
2. `:ssh` calls `CLIHandler.handle_msg({:ssh_channel_up, ...})` → pops pubkey from stash → resolves to a user via `Foglet.Accounts.get_user_by_public_key/1`.
3. `CLIHandler` calls `Sessions.Supervisor.start_session/1` (authenticated) or `start_guest_session/0` (unauthenticated guest).
4. On `:pty` message, `CLIHandler` calls `Raxol.Core.Runtime.Lifecycle.start_link(Foglet.TUI.App, ...)` with a `session_context` map injected via Lifecycle options.
5. `TUI.App.init/1` unpacks `session_context`, sets `current_screen` to `:login` or `:main_menu`, subscribes to PubSub topics via `PubSubForwarder`.
6. Input bytes from SSH → `IOAdapter.parse_input/1` → Raxol Events → dispatched to Dispatcher GenServer → routed to `TUI.App.update/2`.
7. `update/2` dispatches to the active screen's `handle_key/2`, which returns `{:update, new_state, commands}`.
8. `App.process_screen_commands/2` converts I/O dispatch tuples (e.g. `{:load_boards}`) into `Command.task` closures that run off-process and deliver typed result messages back to `update/2`.
9. `TUI.App.view/1` calls the active screen's `render/1` → Raxol renders to ANSI → `CLIHandler`'s CRLF writer → `:ssh_connection.send/3` → terminal.

**Post Creation Flow:**

1. `PostComposer` screen submits → `{:submit_post, body}` → `TUI.App.do_update/2`.
2. `Command.task` closure calls `Foglet.Posts.create_post/4`.
3. `Foglet.Posts` delegates to `Foglet.Boards.Server.create_post/4` (via GenServer call to the board's registered server).
4. `Boards.Server` runs an `Ecto.Multi` that atomically: bumps `boards.next_message_number`, inserts the post with the allocated `message_number`, bumps thread counters, increments `users.post_count`.
5. On success, `Boards.Server` advances its in-memory counter and returns `{:ok, post}`.

**PubSub Propagation:**

Domain contexts broadcast on Phoenix.PubSub after mutations (e.g. `"board:<id>"`, `"thread:<id>"`, `"user:<id>"`). `TUI.App.subscribe/1` returns a `Subscription.custom/2` using `PubSubForwarder`, which subscribes to the relevant topics and forwards arriving messages as `{:subscription, msg}` to the Raxol Dispatcher, which delivers them to `update/2`.

**State Management:**

- Domain state (authoritative): Postgres via Ecto.
- Session-scoped identity: `Foglet.Sessions.Session` GenServer (in-memory struct: `user_id`, `handle`, `role`, `terminal_size`, `tui_pid`, `last_seen_at`).
- UI state: `%Foglet.TUI.App{}` struct held inside the Raxol Lifecycle process.
- Screen-level state: `state.screen_state` map keyed by screen atom (e.g. `state.screen_state[:post_reader]` holds `%PostReader.State{}`).
- Runtime config: ETS table `:foglet_config` (read-through cache over the `configuration` DB table).
- Ephemeral SSH tracking: `PubkeyStash` ETS table, connection count ETS table in `CLIHandler.Counter`.

## Key Abstractions

**`Foglet.TUI.Screen` behaviour** (`lib/foglet_bbs/tui/screen.ex`):
- Purpose: Contract every screen module implements.
- Callbacks: `render/1`, `handle_key/2`, optional `init_screen_state/1`.
- Pattern: `handle_key/2` returns `{:update, new_state, [commands]}` or `:no_match` (bubbles to `TUI.App.global_key_handler/2`).

**`Foglet.TUI.App` — Conductor** (`lib/foglet_bbs/tui/app.ex`):
- Purpose: Top-level Raxol application: owns the `current_screen` routing table, all I/O dispatch command handling, modal lifecycle, PubSub wiring, and session heartbeat.
- Pattern: `use Raxol.Core.Runtime.Application`. All screen modules are stateless pure functions; `App` holds the full model.

**`Foglet.Boards.Server` — Sequence Owner** (`lib/foglet_bbs/boards/server.ex`):
- Purpose: Serializes message-number allocation per board to avoid DB-side sequence complexity.
- Pattern: `use GenServer`, registered via `Foglet.BoardRegistry`. On crash, `init/1` re-seeds from `MAX(message_number)` in the posts table.

**`Foglet.Sessions.Session` — Session Identity** (`lib/foglet_bbs/sessions/session.ex`):
- Purpose: Long-lived per-user GenServer. Starts as a guest (no `user_id`), promoted to authenticated via `promote_to_user/2` after TUI login.
- Pattern: `use GenServer, restart: :temporary`. Named via `{:via, Registry, {Foglet.Sessions.Registry, user_id}}` after promotion.

**`Foglet.Schema`** (`lib/foglet_bbs/schema.ex`):
- Purpose: Shared macro that sets UUID v7 primary keys, UUID foreign keys, and `utc_datetime_usec` timestamps on every Ecto schema.
- Pattern: `use Foglet.Schema` at the top of every schema module.

**`Foglet.Config`** (`lib/foglet_bbs/config.ex`) + **`Foglet.Config.Schema`** (`lib/foglet_bbs/config/schema.ex`):
- Purpose: ETS-cached read-through access to the `configuration` DB table; typed accessors for each key; typed validation before writes.
- Pattern: ETS table `:foglet_config` initialized before the supervisor tree starts. `put!/3` validates against `Config.Schema` before any DB work.

**`Foglet.TUI.PubSubForwarder`** (`lib/foglet_bbs/tui/pub_sub_forwarder.ex`):
- Purpose: Bridge between Phoenix.PubSub and Raxol's update loop. Raxol only routes `{:subscription, msg}` to `update/2`; this GenServer wraps all arriving PubSub messages into that shape.
- Pattern: `Subscription.custom/2` event source — Raxol calls `start_link(args, context)` with `context.pid` as the Dispatcher pid.

## Entry Points

**SSH daemon** (`lib/foglet_bbs/ssh/supervisor.ex`):
- Triggers: Application boot, OTP `:ssh` daemon accepting connections.
- Responsibilities: Asserts OTP >= 27.3.3 (CVE-2025-32433 guard), initializes connection counter ETS, starts `DaemonOwner`.

**Phoenix Endpoint** (`lib/foglet_bbs_web/endpoint.ex`):
- Triggers: Application boot.
- Responsibilities: HTTP server (Bandit), LiveDashboard route (dev/sysop only). No end-user routes.

**`FogletBbs.Application.start/2`** (`lib/foglet_bbs/application.ex`):
- Responsibilities: Initializes `Foglet.Config` ETS cache and `Foglet.SSH.PubkeyStash` ETS before the supervision tree starts. Registers TUI themes. Starts the supervision tree, then calls `Foglet.Boards.boot_board_servers/0` to start one `Boards.Server` per active board.

## Error Handling

**Strategy:** Let it crash (OTP). Supervisors restart workers. Domain errors (`{:error, changeset}`, `{:error, reason}`) are returned explicitly and surfaced to the TUI as error modals.

**Patterns:**
- `Boards.Server` uses `Ecto.Multi` for atomic post/thread creation; on `Multi` failure the message-number counter is NOT advanced (the failed number is reused on the next attempt).
- `Sessions.Supervisor` implements a replacement protocol for the one-session-per-user rule: sends `:replaced_by_new_session`, monitors for `:DOWN`, force-terminates after 2 seconds.
- `TUI.App.do_update({:task_error, op, reason}, state)` catches failed `Command.task` closures and shows a generic error modal.
- `CLIHandler` traps `EXIT` from the Lifecycle process and closes the SSH channel on unexpected exit, preventing hung terminals.

## Cross-Cutting Concerns

**Logging:** `require Logger` + `Logger.info/warning/error` at key lifecycle events (session start/replace, SSH channel up/down, Board Server errors). No structured logging library.

**Validation:** Ecto changesets for all Ecto schemas. `Foglet.Config.Schema.validate/2` for runtime config values before DB writes.

**Authentication:** SSH-level: `no_auth_needed: true` (the `:ssh` daemon accepts all connections; pubkey correlation happens in `CLIHandler` via `PubkeyStash`). Application-level: `Foglet.Accounts.authenticate_by_password/2` (Argon2) and `get_user_by_public_key/1` called from the TUI login screen.

**Theming:** `Foglet.TUI.Theme` registers palettes with Raxol's theme registry on boot and projects them into flat `%Theme{}` structs held in `session_context`. All widget render functions receive a theme parameter rather than reading global state.

**Configuration layering:**
1. Compile-time defaults (`config/config.exs`).
2. Deploy-time env vars (`config/runtime.exs`).
3. Runtime sysop config in the `configuration` DB table, accessed via `Foglet.Config` (ETS-cached).
