## Test Framework

**Runner:** ExUnit (built-in Elixir, no separate version pin)

**Property testing:** `stream_data ~> 1.0` via `ExUnitProperties` — used in `test/foglet_bbs/boards/board_server_test.exs`

**Run Commands:**
```bash
mix test                        # run all tests (also runs ecto.create + ecto.migrate)
mix test path/to/test.exs       # run single file
mix test --failed               # re-run only failures
mix test --exclude pending       # exclude :pending tag (this is the default in test_helper.exs)
```

**Test helper:** `test/test_helper.exs`
```elixir
ExUnit.start(exclude: [:pending])
Ecto.Adapters.SQL.Sandbox.mode(FogletBbs.Repo, :manual)
```

## Test Case Templates

### DataCase — database tests

Use for any test that touches the DB (contexts, schemas, migrations).

```elixir
defmodule Foglet.AccountsTest do
  use FogletBbs.DataCase, async: true   # async: false when sharing global state (ETS, GenServers)
  ...
end
```

`FogletBbs.DataCase` (`test/support/data_case.ex`) sets up the Ecto SQL sandbox and provides:
- `errors_on(changeset)` — converts changeset errors to a `%{field: ["message"]}` map for assertions
- `alias FogletBbs.Repo`, `import Ecto`, `import Ecto.Changeset`, `import Ecto.Query`

### Plain ExUnit.Case — pure logic tests

Use for TUI screen/widget tests and OTP tests that do not touch the DB.

```elixir
defmodule Foglet.TUI.Screens.BoardListTest do
  use ExUnit.Case, async: true
  ...
end
```

## Async vs. Sync Tests

- **`async: true`** — default for DB tests and pure logic tests. Safe when no shared mutable state.
- **`async: false`** — required when:
  - Tests interact with `Foglet.BoardRegistry` or `Foglet.Sessions.Registry` (shared named processes)
  - Tests mutate the `:foglet_config` ETS table (global named table) — e.g., `config_test.exs`, `login_test.exs`
  - Tests start real Board Servers via `Foglet.Boards.Supervisor`

Of 55 test modules that declare async, 40 are `async: true` and 15 are `async: false`.

## Test Structure

**Standard describe/test layout:**
```elixir
describe "function_name/arity (REQUIREMENT-REF)" do
  test "describes expected behavior" do
    # arrange
    user = AccountsFixtures.user_fixture()
    # act
    assert {:ok, result} = Accounts.some_function(user)
    # assert
    assert result.field == expected_value
  end
end
```

**Requirement references in describe strings** — tests are tagged to requirements (e.g., `IDNT-01`, `BOARD-06`, `SSH-05`) to make traceability explicit.

## Fixtures

Two fixture modules live in `test/support/`:

### `FogletBbs.AccountsFixtures` (`test/support/accounts_fixtures.ex`)

```elixir
# Valid attrs with unique values — override any key
AccountsFixtures.valid_user_attributes(%{handle: "alice"})

# Insert a user via Foglet.Accounts.register_user/1
AccountsFixtures.user_fixture(%{password: "mypassword"})

# Insert an SSH key for a user
AccountsFixtures.ssh_key_fixture(user, %{public_key: "ssh-ed25519 ..."})

# Build and insert an email token
AccountsFixtures.user_token_fixture(user, "confirm")  # or "reset_password"
```

Uniqueness is guaranteed with `System.unique_integer([:positive])` embedded in handle and email.

### `FogletBbs.BoardsFixtures` (`test/support/boards_fixtures.ex`)

```elixir
# Create domain objects
BoardsFixtures.category_fixture(%{name: "Tech"})
BoardsFixtures.board_fixture(category, %{slug: "tech"})
BoardsFixtures.thread_fixture(board, user, %{title: "My Thread"})
BoardsFixtures.post_fixture(thread, user, %{body: "Reply body"})

# Valid attrs builders (for testing changeset validation separately)
BoardsFixtures.valid_board_attributes(%{slug: "custom"})
BoardsFixtures.valid_thread_attributes()
BoardsFixtures.valid_post_attributes()
```

`BoardsFixtures` also includes its own `user_fixture/1` (duplicating the one in `AccountsFixtures`).

## OTP Test Patterns

### Starting supervised processes

Always use `start_supervised!/1` — guarantees cleanup between tests.

```elixir
pid = start_supervised!({Foglet.Sessions.Session, [user_id: user_id, handle: "alice", role: :user]})
```

For Board Servers, pass a unique `id:` to allow multiple servers in one test:
```elixir
{pid, sup_id} = start_supervised!({Server, board_id: board.id}, id: {:board_server, board_id, :v1})
```

### Synchronizing with process state

Use `:sys.get_state/1` to flush the message queue before asserting on state:
```elixir
:ok = Session.heartbeat(user_id)
_ = :sys.get_state(Session.via_tuple(user_id))   # flush cast
state = Session.get_state(user_id)
assert DateTime.compare(state.last_seen_at, initial) in [:eq, :gt]
```

### Waiting for process termination

Monitor and assert on `:DOWN`, never `Process.alive?`:
```elixir
ref = Process.monitor(pid)
send(pid, :replaced_by_new_session)
assert_receive {:DOWN, ^ref, :process, ^pid, :normal}, 1_000
```

### Sandbox access for GenServer tests

When a GenServer holds a DB connection (Board Server, etc.), allow the sandbox:
```elixir
pid = start_supervised!({Server, board_id: board.id})
Ecto.Adapters.SQL.Sandbox.allow(Repo, self(), pid)
```

## TUI Widget Test Patterns

Widget tests use `use ExUnit.Case, async: true` and import shared helpers:

```elixir
import Foglet.TUI.WidgetHelpers,
  only: [flatten_text: 1, color_atom_leaked?: 2, color_names: 0]
```

**`Foglet.TUI.WidgetHelpers`** (`test/support/foglet/tui/widget_helpers.ex`) provides:
- `flatten_text/1` — walks a Raxol render tree and joins all `:content`/`:text` leaf values into a single string for substring assertions
- `color_atom_leaked?/2` — checks whether a raw color atom (`:red`, `:green`, etc.) leaked into a render tree using word-boundary regex (avoids false positives on `:hovered_red`)
- `color_names/0` — returns the list of 8 core color names checked for hygiene

**Standard widget test groups:**
1. `describe "init/1"` — initializes widget state correctly
2. `describe "handle_event/2"` — input events produce correct state transitions and action returns
3. `describe "render/2 — smoke"` — render returns a non-nil map with `:type` key; no crash
4. `describe "render/2 — theme hygiene"` — no hardcoded color atoms leak; alt-theme produces different output
5. `describe "render/2 — option variants"` — e.g., `bordered: true/false`

Render assertions inspect the serialized tree rather than structural matching:
```elixir
serialized = inspect(result, printable_limit: :infinity, limit: :infinity)
assert serialized =~ "mytext"
refute serialized =~ ~s(type: :box)
```

## TUI Screen Test Patterns

Screen tests construct minimal `Foglet.TUI.App` struct state directly — no real app process:

```elixir
defp base_state do
  %Foglet.TUI.App{
    current_screen: :login,
    current_user: nil,
    session_context: %{registration_mode: "open"},
    terminal_size: {80, 24},
    screen_state: %{}
  }
  |> Map.from_struct()
end
```

Inline fake domain modules mock out DB calls in pure screen tests:
```elixir
defmodule FakeBoards do
  def list_subscribed_boards(_user) do
    [%{id: "b1", name: "General", slug: "general", unread_count: 3}]
  end
end
```

Screen function return shapes:
- `{:update, new_state, commands}` — state transition with command list
- `:no_match` — key not handled by this screen

## ETS Cache Isolation in Tests

Tests that exercise `Foglet.Config` must manage the `:foglet_config` ETS table. Patterns used:

**Full isolation (config_test.exs):**
```elixir
setup do
  Config.init_cache()
  for key <- @test_keys, do: Config.invalidate(key)
  on_exit(fn -> for key <- @test_keys, do: Config.invalidate(key) end)
  :ok
end
```

**Selective restore (login_test.exs):**
```elixir
setup do
  original = Foglet.Config.get("require_email_verification", :not_seeded)
  on_exit(fn ->
    case original do
      :not_seeded -> :ok
      value -> Foglet.Config.put!("require_email_verification", value)
    end
  end)
  :ok
end
```

**ETS seeding for pure tests (app_test.exs):**
```elixir
setup do
  Config.init_cache()
  :ets.insert(:foglet_config, {"registration_mode", "open"})
  :ets.insert(:foglet_config, {"email_verify_resend_cooldown_seconds", 60})
  :ok
end
```

## Coverage Approach

No enforced coverage threshold. Coverage is assessed qualitatively:

**Well covered:**
- All context modules (`Foglet.Accounts`, `Foglet.Boards`, `Foglet.Posts`, `Foglet.Threads`)
- All schema changeset pathways (valid and invalid attrs)
- `Foglet.Config` full API including ETS cache, typed accessors, and validation errors
- OTP processes (`Foglet.Boards.Server`, `Foglet.Sessions.Session`, `Foglet.Sessions.Supervisor`)
- SSH key registration and fingerprint verification

**Notable coverage gaps:**

- `lib/foglet_bbs/time_ago.ex` — not confirmed to have a test file

## What NOT to Use in Tests

- `Process.sleep/1` — banned; synchronize with `:sys.get_state/1` or `assert_receive` + `:DOWN` instead
- `Process.alive?/1` — banned; use `:DOWN` monitoring instead
- Raw `start_link/1` for OTP processes — use `start_supervised!/1` always
- `mix deps.clean --all` to fix test issues — diagnose the root cause instead
