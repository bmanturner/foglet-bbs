<objective>
Refactor `Foglet.TUI.Screens.MainMenu` data layer to expose two canonical, mutually-exclusive lists derived from a SINGLE source-of-truth descriptor list `@main_menu_commands`. Each entry carries a `:kind` tag (`:destination` or `:action`); `visible_destinations/1` returns destination entries filtered by ShellVisibility, and `visible_actions/1` returns action entries gated by current_user / Bodyguard.permit? / non-empty oneliners. The structural disjointness of destinations vs. actions becomes a DATA property (one list, two filters) instead of an enforced-by-tests convention. This plan does NOT change the rendered visual yet — the body still uses the existing `column` layout. Plan 02 layers the boxed Navigation/Oneliners panels on top of this data layer; Plan 03 layers the size-contract assertions.

Purpose: D-01 mandates a single source of truth for "what is a destination vs. action". The single `@main_menu_commands` list with a `:kind` tag is the structural realization of that mandate — destinations and actions cannot drift because they are two filtered views of one list. Doing the data layer in isolation also keeps tests green between waves.

Output: Refactored `main_menu.ex` with a single `@main_menu_commands` descriptor list and public `visible_destinations/1` / `visible_actions/1` derived by filtering it; extended `main_menu_test.exs` covering role visibility for the destinations list, structural disjointness as a property assertion, literal B/C/A/M/S/Q non-duplication invariant, command-bar action visibility (O/H/↑↓), and anonymous-`C` destination presence.
</objective>

    ```elixir
    @spec visible_destinations(map() | nil) :: [{String.t(), String.t()}]
    def visible_destinations(user) do
      # Build a minimal state shim so the shared gate function can run; destination
      # visibility never depends on oneliner state, so the shim has no oneliners.
      state = %{current_user: user, recent_oneliners: []}

      @main_menu_commands
      |> Enum.filter(&(&1.kind == :destination))
      |> Enum.filter(&command_visible?(&1.visibility, user, state))
      |> Enum.map(&{&1.key, &1.label})
    end
    ```

    **Add `visible_actions/1` (NEW — replaces `visible_menu_keys/1`):**
    Public function. Filters the shared descriptor list to `:action` entries that pass the visibility gate, then groups them into command-bar groups for `ScreenFrame.render/4`.
    ```elixir
    @spec visible_actions(map()) :: [%{label: String.t(), commands: [map()]}]
    def visible_actions(state) do
      user = state.current_user

      visible =
        @main_menu_commands
        |> Enum.filter(&(&1.kind == :action))
        |> Enum.filter(&command_visible?(&1.visibility, user, state))

      hide_oneliner = Enum.filter(visible, &(&1.key == "H")) |> Enum.map(&{&1.key, &1.label})
      oneliner_post = Enum.filter(visible, &(&1.key == "O")) |> Enum.map(&{&1.key, &1.label})
      select_oneliner = Enum.filter(visible, &(&1.key == "↑/↓")) |> Enum.map(&{&1.key, &1.label})

      [
        command_group("Actions", hide_oneliner ++ oneliner_post, 10),
        command_group("Select", select_oneliner, 20)
      ]
      |> Enum.reject(&(&1.commands == []))
    end
    ```
    Note the explicit "Select" group per RESEARCH.md Open Question 2 recommendation — keeps the three actions logically grouped instead of letting `Normalizer` reroute `↑/↓` into a synthesized "Navigate" group. The `:hide_oneliner` action ordering inside "Actions" puts H before O so the negative `command_priority("H", _) -> -10` lifts H to the front of the displayed group (matching existing priority ordering for H).
