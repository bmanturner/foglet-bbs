<objective>
Close BL-02: persist `submit_state` on `Foglet.TUI.Screens.Sysop.SiteForm.State`
so the FORM-05 lock guard and the FORM-05 status-row contract (D-08, D-09)
survive across the per-render Modal.Form rebuild.

Today, `SiteForm.render/2` (site_form.ex:60-65) and the catch-all
`SiteForm.handle_key/2` clause (site_form.ex:78-92) both call
`SState.build_modal_form/1`, which always returns a fresh form with
`submit_state: :idle` (state.ex:122-138). Even though `persist_payload/3`
(site_form.ex:180-183) sets `submit_state: :saved` and `finalize_submit/2`
(site_form.ex:124) sets `{:error, "validation"}`, `sync_back/2`
(site_form.ex:189-193) writes back ONLY `drafts` and `focused` — the form's
`submit_state` is discarded on every event. Effects: (a) FORM-05 lock has
zero effect on this consumer, double-Enter / double-Ctrl+S calls
`Foglet.Config.put/3` twice, (b) the operator never sees "Saved." or
"Error: validation" status rows on the SITE form.

Fix shape (per 28-REVIEW.md BL-02 option 2 — minimal, additive):
  - Add `submit_state: :idle` to the SState defstruct.
  - In `render/2` and the catch-all `handle_key/2` clause, re-apply
    `state.submit_state` to the freshly-built Modal.Form via
    `Map.put(form, :submit_state, state.submit_state)` before
    `apply_errors` and dispatch.
  - In `sync_back/2`, persist the form's `submit_state` back onto SState.
  - In `persist_payload/3` success branch, set SState.submit_state = :saved
    BEFORE calling sync_back so the persisted value lands on SState.
  - In `finalize_submit/2` validation-failure branch, set
    SState.submit_state = {:error, "validation"} similarly.

Output: SiteForm preserves submit_state across renders; new tests assert
double-Enter single-fire, "Saved." appears, and "Error: validation" appears.
</objective>
