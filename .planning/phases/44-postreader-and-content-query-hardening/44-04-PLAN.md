<objective>
Protect the soft-delete visibility split with focused tests and, where useful, shared query helper usage: reader/history keeps tombstones while list and summary paths hide deleted content.
</objective>

<must_haves>
<truths>
- POST-04: Maintainers need automated coverage or a shared query helper preventing soft-deleted posts from reappearing in list paths.
- D-13: Reader/history queries are tombstone-capable.
- D-14: `Threads.list_threads/1`, `Threads.list_threads/2`, board directory unread/last-post summaries, and `QueryHelpers.not_deleted/1` paths need explicit protection.
- D-15: Prefer `Foglet.QueryHelpers` or context-owned query helpers over ad hoc deleted-row predicates in screens.
- Phase boundary: do not hide soft-deleted posts from PostReader/history views entirely.
</truths>
</must_haves>

<threat_model>
- T-44-10 (medium): Adding broad `not_deleted/1` to `Foglet.Posts.list_posts/1` would break reader history. Mitigation: tests assert both `list_posts/1` and `list_reader_window/2` include soft-deleted rows.
- T-44-11 (medium): Thread/board summaries may count deleted posts because joins use local predicates that drift. Mitigation: add explicit tests around deleted posts and deleted threads for every listed path.
- T-44-12 (low): Moving predicates without tests can alter unread counts for subscribed boards. Mitigation: assert unread counts and directory entries after soft deletion.
</threat_model>
