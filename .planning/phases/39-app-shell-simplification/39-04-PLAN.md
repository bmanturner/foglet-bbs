<objective>
Five screens — MainMenu, Moderation, Sysop, ThreadList, PostReader — gain a new `update(:on_route_enter, state, ctx)` clause. The clauses preserve the conditional-load semantics that App's per-screen `maybe_dispatch_route_entry/3` clauses currently encode (`app.ex:810-845`), but invert ownership: the screen, not App, decides whether to load.

This is the "screen-side groundwork" for Plan 39-05's App-side cleanup. After this plan, App's per-screen clauses still exist and still trigger the right loads — but the screens now ALSO respond to a generic `:on_route_enter` message, so Plan 39-05 can collapse the five App clauses into one screen-agnostic clause without losing any behavior.

Per CONTEXT.md "Claude's Discretion": each screen's `:on_route_enter` clause delegates to its existing `:load`/`:load_oneliners` clause rather than reimplementing the load logic, which preserves the existing test seam (direct `update(:load, …)` callers in reducer tests keep working) and produces the smallest diff.

Purpose: D-01, D-03, D-14. Without these clauses, deleting the per-screen App clauses in Plan 39-05 would break first-load behavior for these five screens.

Output: Five screen modules each gain one new `update/3` clause inserted above their existing `:load` (or `:load_oneliners`) clause. No deletions in this plan; deletions are wave 2.
</objective>
