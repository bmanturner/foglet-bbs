## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] MultiLineInput.update/2 returns {:noreply, state, cmds} tuple**
- **Found during:** Task 1, textarea dispatch
- **Issue:** Plan spec assumed `MultiLineInput.update/2` returns the new state directly; actual return is `{:noreply, %MultiLineInput{}, nil}`
- **Fix:** Pattern-match `{:noreply, new_state, _cmds}` in dispatch_to_field with fallback for bare state
- **Files modified:** lib/foglet_bbs/tui/widgets/modal/form.ex

**2. [Rule 1 - Bug] MultiLineInput value/lines desync after enter + subsequent edits**
- **Found during:** Task 1, textarea coercion test
- **Issue:** After `{:enter}` message, `value` field becomes `"h\n"` but `lines` stays `["h"]`. After further input, `value` reverts to just `"h"` (drops the newline). Joining `lines` with `"\n"` also incorrect.
- **Fix:** Maintain a `raw_value` string in the textarea field state map, updated by `apply_raw_edit/2` on each event. Coerce from `raw_value` rather than from MultiLineInput state.
- **Files modified:** lib/foglet_bbs/tui/widgets/modal/form.ex

**3. [Rule 1 - Bug] `%{key: :char, char: "\n"}` filtered by Compose.translate_key**
- **Found during:** Task 1, textarea newline test
- **Issue:** `String.codepoints("hello\nworld")` produces a `%{key: :char, char: "\n"}` event. `Compose.translate_key` filters codepoints < 32, returning nil.
- **Fix:** In textarea dispatch, pre-check for `\n` char and map it to `{:enter}` before calling `Compose.translate_key`.
- **Files modified:** lib/foglet_bbs/tui/widgets/modal/form.ex
