## Warnings

### WR-01: Enum dispatch clamps to `-1` when `choices` list is empty

**File:** `lib/foglet_bbs/tui/widgets/modal/form.ex:231`

**Issue:** The `:down` handler computes `min(field_state + 1, length(choices) - 1)`.
When `choices` is an empty list `length([]) - 1 = -1`, so pressing Down on an
empty enum field sets the index to `-1`. `Enum.at/2` with a negative index counts
from the tail in Elixir, so a subsequent submit will coerce to `Enum.at([], -1)`
which is `nil` — silently producing `nil` instead of a choice atom. Worse, if
some future caller passes a non-empty list and the index is somehow already
negative (e.g., a pre-seeded value) the same `min/2` guard won't save it.

The `:up` clause (line 234) is fine because `max(idx - 1, 0)` floors at 0.

**Fix:** Guard the `:down` clause so it is a no-op when choices is empty, and
clamp the floor at 0 so a stale negative index can't survive:

```elixir
defp dispatch_to_field(%{type: :enum, choices: []}, field_state, %{key: :down}),
  do: field_state

defp dispatch_to_field(%{type: :enum, choices: choices}, field_state, %{key: :down}) do
  max(0, min(field_state + 1, length(choices) - 1))
end
```

The `max(0, ...)` wrapper is cheap insurance against any pre-seeded negative index
surviving into the render/coerce path.

### WR-02: `:text` coerce clause can return `nil`, violating the `String.t()` spec

**File:** `lib/foglet_bbs/tui/widgets/modal/form.ex:280`

**Issue:** `coerce(%{type: :text}, %TextInput{raxol_state: %{value: v}})` binds `v`
directly from the vendor struct and returns it as-is. If `TextInput.raxol_state.value`
is ever `nil` (possible if the vendor changes the default initialisation, or if a
caller constructs a `TextInput` outside `build_field_state`), the submit payload
receives `nil` for a `:text` field. The module spec declares `:text` → `String.t()`,
so callers are entitled to trust that guarantee.

The integer coerce clause at line 283 already defensively handles this via
`String.trim(v || "")`, but the text clause does not.

**Fix:** Mirror the same nil-coalesce guard:

```elixir
defp coerce(%{type: :text}, %TextInput{raxol_state: %{value: v}}), do: v || ""
```

This is a one-character change that closes the gap between the declared spec and
the actual return type.

## Info

### IN-01: Double `String.length/1` call in `apply_raw_edit/2` backspace clause

**File:** `lib/foglet_bbs/tui/widgets/modal/form.ex:341-346`

**Issue:** The backspace clause calls `String.length(rv)` twice — once to check
for empty and again as the slice bound. `String.length/1` is O(n) on the number
of Unicode graphemes, so this is two full traversals of the string per keystroke.
It also reads slightly more verbosely than the idiomatic range-slice form.

**Fix:** Use a range slice that already handles the empty case correctly (returns
`""` when the string is empty):

```elixir
defp apply_raw_edit(rv, %{key: :backspace}), do: String.slice(rv, 0..-2//1)
```

`String.slice("", 0..-2//1)` returns `""`, so the length guard is not needed.
This collapses four lines to one and makes the intent immediately obvious.
