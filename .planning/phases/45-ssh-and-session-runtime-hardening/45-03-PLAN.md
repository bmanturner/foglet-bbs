<objective>
Centralize SSH channel cleanup in `Foglet.SSH.CLIHandler` and prove the global connection counter stays balanced across all roadmap-listed lifecycle and rejection paths.
</objective>

<must_haves>
<truths>
- SSH-03: Maintainer can change SSH channel termination behavior through one helper that owns alt-screen leave, lifecycle stop, session stop, and connection-count cleanup.
- SSH-04: Operator can trust the global SSH connection counter after normal closes, EOF-to-close, lifecycle exits, over-limit rejects, rate-limit rejects, and crash-during-init paths.
- D-07: One cleanup helper owns alt-screen leave, lifecycle stop, session stop, connection-counter decrement, and channel close where applicable.
- D-08: Cleanup must be idempotent for EOF, closed, lifecycle EXIT, and terminate callback ordering.
- D-09: Terminal restoration stays best-effort and direct to the SSH channel.
- D-10: Direct callback/unit tests are acceptable for proving counter balance.
- D-11: Tests reset the ETS counter explicitly and assert expected values without upward drift or negative values.
- D-12: Over-limit and rate-limit rejects are rejected states that must not require later decrement.
</truths>
</must_haves>

<threat_model>
- T-45-06 (high): Multiple callbacks can decrement the active connection counter more than once. Mitigation: add state flags such as `cleanup_done?` and `counter_counted?`, and make cleanup idempotent.
- T-45-07 (high): Rejection paths can leak or double-decrement counter state. Mitigation: tests explicitly cover over-limit and rate-limit rejection with no later cleanup dependency.
- T-45-08 (medium): Central cleanup can close the channel before alt-screen leave reaches the client. Mitigation: helper sends alt-screen leave before optional channel close.
</threat_model>

<action>
Refactor `Foglet.SSH.CLIHandler` cleanup per D-07 through D-09:

- Add `cleanup_done?: false` and `counter_counted?: false` to the struct.
- When `check_connection_limit/0` returns `:ok` and rate limiting allows the peer, set `counter_counted?: true` on the returned state. Do not call a separate increment helper.
- For over-limit and rate-limit reject states, set `over_limit: true`, `cleanup_done?: true`, and `counter_counted?: false` after the immediate reject/compensation path.
- Add one private helper, named `cleanup/2` or similarly, that performs in this order: `send_alt_screen_leave(state)`, `stop_lifecycle(state.lifecycle_pid)`, `stop_session(state.session_pid)`, optional `maybe_close_channel(state)`, and counter decrement only when `state.counter_counted?` is true.
- The helper must return an updated state with `cleanup_done?: true` and `counter_counted?: false`; if called again on a cleaned state it returns the state without repeating side effects.
- Change lifecycle `{:EXIT, lifecycle_pid, reason}`, `{:closed, _}`, and `terminate/2` to delegate cleanup-sensitive behavior to this helper.
- Keep `{:eof, _}` as alt-screen leave only unless the implementation can prove it will not double-close or double-decrement before `{:closed, _}`.
</action>
