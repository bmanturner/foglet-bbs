<objective>
Carry SSH peer context into guest-to-user promotion and emit structured promotion audit metadata while preserving existing one-session-per-user and forced replacement fallback behavior.
</objective>

<must_haves>
<truths>
- SSH-02: Operator has audit-grade visibility into guest-to-user SSH session promotions, including peer context where available.
- SESS-01: Maintainer has direct coverage for the `replace_then_promote/3` forced-termination fallback.
- D-04: Peer metadata must move from `CLIHandler` into the session context/promotion path.
- D-05: Guest promotion logging must include promoted session identity, target user id/handle, peer context when available, and replacement context when known.
- D-06: Preserve one-session-per-user semantics and the Registry-slot-before-promotion invariant.
- D-13: Preserve the existing deterministic forced fallback test unless equivalent proof is needed.
- D-14: Keep fallback proof synchronized with monitors and Registry/state checks.
</truths>
</must_haves>

<threat_model>
- T-45-03 (medium): Promotion audit can omit peer context because login happens after channel-up. Mitigation: add `ssh_peer` to `SessionContext` and pass it through `App` to the session supervisor.
- T-45-04 (high): Audit changes can accidentally promote before the old session releases its Registry slot. Mitigation: keep replacement decisions in `Sessions.Supervisor` and pass audit metadata only after replacement handling.
- T-45-05 (medium): Log tests can become brittle if they only match strings. Mitigation: assert state/Registry behavior directly and use focused log capture only for the promotion audit boundary.
</threat_model>
