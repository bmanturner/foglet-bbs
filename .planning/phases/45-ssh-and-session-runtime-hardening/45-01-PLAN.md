<objective>
Bound SSH public-key offer handoff state by adding timestamped TTL/sweep behavior to `Foglet.SSH.PubkeyStash` without changing `KeyCB` or `CLIHandler` guest fallback semantics.
</objective>

<must_haves>
<truths>
- SSH-01: Operator has bounded SSH public-key stash behavior through TTL or sweep cleanup for orphaned key-offer entries.
- D-01: TTL/sweep behavior belongs inside `Foglet.SSH.PubkeyStash`; no durable storage and no SSH authentication model change.
- D-02: Keep `put/2` and `pop/1` compatible for `KeyCB` and `CLIHandler`; add only the smallest deterministic test API.
- D-03: Missing or expired stash entries still become guest sessions.
</truths>
</must_haves>

<threat_model>
- T-45-01 (medium): A stale key offer can be consumed by a later connection from the same peer tuple. Mitigation: `pop/2` deletes expired entries and returns `:miss`.
- T-45-02 (low): TTL metadata can break current tuple shape assumptions. Mitigation: keep public `put/2` and `pop/1` arities compatible and update focused stash tests.
</threat_model>

<action>
Update `lib/foglet_bbs/ssh/pubkey_stash.ex` per D-01 through D-03:

- Replace the module doc TTL section with the exact behavior: entries are timestamped on `put/2`, `pop/1` treats expired entries as `:miss`, and `sweep/2` removes stale orphan entries.
- Add `@ttl_ms :timer.minutes(5)`.
- Keep `put(peer_key, public_key)` and implement it as `put(peer_key, public_key, System.monotonic_time(:millisecond))`.
- Add `put(peer_key, public_key, now_ms)` for deterministic tests. It must store `{peer_key, public_key, now_ms}`.
- Keep `pop(:unknown)`, `pop(peer_key)`, and add `pop(peer_key, now_ms)`.
- `pop(peer_key, now_ms)` must use `:ets.take/2`; for `{peer_key, public_key, inserted_at_ms}`, return `{:ok, public_key}` only when `now_ms - inserted_at_ms <= @ttl_ms`; otherwise return `:miss`.
- Add `sweep(now_ms \\ System.monotonic_time(:millisecond), ttl_ms \\ @ttl_ms)` that deletes only entries where `now_ms - inserted_at_ms > ttl_ms` and returns the number of deleted entries.
- If a legacy two-tuple `{peer_key, public_key}` is encountered, `pop/2` may return `{:ok, public_key}` so rolling test setup cannot strand compatible entries.
</action>
