<objective>
Extract App routing, screen-state, context construction, screen module resolution, route initialization, route-entry dispatch, reducer routing, and screen rendering into `Foglet.TUI.App.Routing`.
</objective>

<must_haves>
<truths>
- TUI-04: Maintainer can understand and modify `Foglet.TUI.App` through narrow runtime helper modules for routing, modal, subscription, and effect concerns.
- D-01: The routing helper module is `Foglet.TUI.App.Routing`.
- D-02: `Foglet.TUI.App.Routing` is an App-shell runtime helper that may operate on `%Foglet.TUI.App{}` state but must not own durable domain mutations or screen-specific business workflows.
- D-03: `Foglet.TUI.App` remains the Raxol callback integration point and owns when to call routing.
- D-04: Do not retain obsolete public `Foglet.TUI.App` helper functions only because legacy tests call them.
- D-05: Move behavior coverage to helper-level tests when routing becomes the owner; keep App-level tests for callback integration.
- D-06: Any App delegator that remains for routing must have a production/public-boundary reason.
- D-07: Extract routing first because modal submit, effect interpretation, rendering, and subscriptions depend on it.
- D-11: Routing owns route encoding, screen-key derivation, current/local screen-state reads and writes, screen module resolution, `domain.screen_modules` override fallback, context construction, route-owned screen initialization, route-entry dispatch, and reducer routing.
- D-12: Preserve route-owned reinitialization for `:thread_list`, `:post_reader`, `:post_composer`, and `:new_thread`.
- D-13: Invalid `domain.screen_modules` override atoms remain visible through logging and fall back to built-in screen resolution.
- D-23: Route contract assertions move from broad App tests into focused routing-helper tests while App integration tests still cover callback wiring.
- D-24: Tests assert behavior and state/command/message outcomes, not text presence.
</truths>
</must_haves>

- `route_screen_update/3` calls the target module's `update/3` only when `Code.ensure_loaded?(module)` and `function_exported?(module, :update, 3)` are true.
- `render_screen/1` calls the target module's `render/2` when available, otherwise logs and returns `text("")`.
