<objective>
Extract stable runtime subscriptions, user topics, screen-declared topics, initial route-enter wiring, and dynamic PubSub topic refresh into `Foglet.TUI.App.Subscriptions`.
</objective>

<must_haves>
<truths>
- TUI-04: Maintainer can understand and modify `Foglet.TUI.App` through narrow runtime helper modules for routing, modal, subscription, and effect concerns.
- D-01: The subscription helper module is `Foglet.TUI.App.Subscriptions`.
- D-02: `Foglet.TUI.App.Subscriptions` is an App-shell runtime helper that may operate on `%Foglet.TUI.App{}` state but must not own durable domain mutations or screen-specific business workflows.
- D-03: `Foglet.TUI.App` remains the Raxol callback integration point and owns when to call subscription runtime behavior.
- D-04: Do not keep App subscription helpers solely for old tests.
- D-05: Move subscription coverage to helper-level tests while App-level tests cover callback integration.
- D-06: Any App subscription delegator that remains must have a production/public-boundary reason.
- D-10: Extract subscriptions after routing is stable because screen topics depend on screen keys, screen modules, local screen state, and `Context` construction.
- D-20: Subscriptions owns heartbeat gating by `session_pid`, chrome clock intervals, `PubSubForwarder` wiring, `InitialRouteEnterForwarder` wiring, user topics, screen `subscriptions/2` delegation, and dynamic topic refresh.
- D-21: Preserve `PubSubForwarder.refresh/1` only-when-effective-topic-list-changes behavior.
- D-22: Preserve initial route-enter subscription behavior so authenticated sessions mounted directly by `App.init/1` still receive exactly one `:on_route_enter` pass after dispatcher start.
- D-23: Subscription contract assertions move to focused helper tests while App integration tests retain `subscribe/1` wiring.
- D-24: Tests assert subscription structs and refresh effects, not text presence.
</truths>
</must_haves>

Create `lib/foglet_bbs/tui/app/subscriptions.ex` defining `defmodule Foglet.TUI.App.Subscriptions`.

Implement this public API:

- `subscribe/1`
- `topics/1`
- `screen_declared_topics/1`
- `refresh_dynamic/2`

Move the current subscription behavior from App:

- `subscribe/1` returns heartbeat interval only when `is_pid(state.session_pid)`.
- `subscribe/1` always includes the chrome/main-menu clock interval for `:main_menu_clock_tick`.
- `subscribe/1` includes `Subscription.custom(PubSubForwarder, %{topics: topics(state)})`.
- `subscribe/1` includes `Subscription.custom(InitialRouteEnterForwarder, %{})`.
- `topics/1` returns `[PubSub.user_topic(state.current_user.id)]` when `state.current_user` exists, plus `screen_declared_topics/1`.
- `screen_declared_topics/1` resolves the active screen through `Routing.screen_key(Routing.current_route(state))` and `Routing.screen_module_for(state, key)`, then calls `module.subscriptions(Routing.screen_state_for(state, key), Routing.build_context(state))` only when exported.
- `refresh_dynamic/2` computes old and new topics and calls `PubSubForwarder.refresh(new_topics)` only when `old_topics != new_topics`.
</action>
