<objective>
Extract modal overlay rendering, modal key precedence, modal dismissal, confirm callbacks, form submit routing, and generic form-submit failure visibility into `Foglet.TUI.App.Modal`.
</objective>

<must_haves>
<truths>
- TUI-04: Maintainer can understand and modify `Foglet.TUI.App` through narrow runtime helper modules for routing, modal, subscription, and effect concerns.
- D-01: The modal helper module is `Foglet.TUI.App.Modal`.
- D-02: `Foglet.TUI.App.Modal` is an App-shell runtime helper that may operate on `%Foglet.TUI.App{}` state but must not own durable domain mutations or screen-specific business workflows.
- D-03: `Foglet.TUI.App` remains the Raxol callback integration point and owns when to call modal runtime behavior.
- D-04: Do not keep App modal helper functions solely for old tests.
- D-05: Move modal ownership coverage to helper-level tests while App-level tests cover shell integration.
- D-06: Any thin App modal delegator must have a production/public-boundary reason.
- D-08: Extract modal runtime after routing so form submit can deliver reducer messages through routing.
- D-14: Modal owns overlay rendering, modal key precedence, confirm yes/no callbacks, info/error/warning dismissal keys, form event routing, and generic form-submit failure visibility.
- D-15: Modal may call routing/effect callbacks supplied by App or helper modules, but must not learn screen-specific business logic.
- D-16: Missing or invalid modal-submit targets still produce a visible error modal, not a silent no-op.
- D-23: Modal contract assertions move to focused helper tests while App integration tests retain modal/effect round trips.
- D-24: Tests assert state/commands/modal structs, not rendered text presence.
</truths>
</must_haves>

<action>
Create `lib/foglet_bbs/tui/app/modal.ex` defining `defmodule Foglet.TUI.App.Modal`.

Implement this public API:

- `render_overlay/2`
- `handle_key/2`
- `dismiss/1`
- `confirm/2`
- `submit_error/1`

Move the current modal behavior from App:

- `render_overlay(modal, state)` uses `Theme.from_state(state)`, centers a double-border box, and renders `Widgets.Modal.render(modal, theme)`.
- `handle_key(key_event, state)` dispatches by `state.modal.type`.
- Confirm modal key behavior: `%{key: :char, char: "y" | "Y"}` calls `confirm(state, :yes)`, `%{key: :char, char: "n" | "N"}` and `%{key: :escape}` call `confirm(state, :no)`.
- Info/error/warning behavior: `%{key: :enter}`, `%{key: :escape}`, and `%{key: :char, char: " "}` dismiss.
- Form behavior: call `Foglet.TUI.Widgets.Modal.Form.handle_event(key, form)`, update `state.modal.message` to the returned form, then route `{:submitted, %Effect{type: :modal_submit}}` through the existing effect/routing path. Route `:submitted` and `{:submitted, _other}` to `submit_error/1`; route `:cancelled` to `dismiss/1`.
- `submit_error/1` sets `%Foglet.TUI.Modal{type: :error, title: "Form Error", message: "Unable to submit form."}`.
</action>
