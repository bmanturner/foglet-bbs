<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| TUI/SSH -> Accounts | User-submitted labels, public keys, and offered SSH keys cross into persistence and authentication logic. |
| Accounts -> Postgres | Durable key rows, uniqueness, ownership, deletion, and last-used metadata are written. |
| SSH daemon -> CLIHandler | Offered key material from the SSH callback is encoded and passed to Accounts. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-12-01 | Elevation of Privilege | `Accounts.revoke_ssh_key/2` | mitigate | Query by both `id` and `user_id`; cross-user revoke returns `{:error, :not_found}` with no delete. |
| T-12-02 | Tampering | duplicate key registration | mitigate | Keep DB unique indexes and `unique_constraint/3` for global fingerprint and per-user label uniqueness. |
| T-12-03 | Spoofing | `Accounts.authenticate_by_public_key/1` | mitigate | Resolve only fingerprints backed by an existing key row joined to a non-deleted user. |
| T-12-04 | Repudiation | `ssh_keys.last_used_at` | mitigate | Update `last_used_at` only after successful registered-key match; tests assert failed/password/guest paths do not update metadata. |
| T-12-05 | Information Disclosure | auth failure return shape | mitigate | Return `{:error, :not_found}` for invalid, unregistered, revoked, and deleted-user keys to avoid differentiating failure causes. |
</threat_model>
