<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator shell -> Mix task | Operator-provided handle enters break-glass reset retrieval. |
| Mix task -> Accounts context | Task requests reset-token generation through the domain boundary. |
| Accounts -> database | Hashed reset-token row is persisted in `user_tokens`. |
| operator output -> user handoff | Raw token is disclosed once through an operator-controlled channel. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-15-01 | Information Disclosure | `Mix.Tasks.Foglet.User.ResetPassword` | mitigate | Remove browser URLs and print only `Reset token: TOKEN`; tests must reject `http://`, `https://`, and `/users/reset_password/`. |
| T-15-02 | Information Disclosure | `Foglet.Accounts.request_password_reset_delivery/1` | mitigate | Do not change user-facing generic response semantics; Plan 1 edits only operator helper/task behavior. |
| T-15-03 | Elevation of Privilege | `Foglet.Accounts.UserToken` | mitigate | Use `UserToken.build_email_token/2` and hashed DB storage; do not hand-roll token generation. |
| T-15-04 | Elevation of Privilege | reset-token replay | mitigate | Preserve `reset_user_password/2` token deletion semantics; this plan does not create a token-consumption path. |
| T-15-05 | Elevation of Privilege | deleted-user reset | mitigate | Keep deleted-user rejection in the Mix task and tests. |
| T-15-06 | Spoofing | unsupported route/product-surface claims | mitigate | Do not add or reference `/users/reset_password`; grep acceptance criteria block route and copy claims. |
</threat_model>
