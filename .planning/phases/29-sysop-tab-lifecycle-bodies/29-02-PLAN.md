<objective>
On top of Plan 01's lifecycle foundation, deliver the operator-facing
behaviors that complete SYSOP-02 (retry + forbidden distinction) and
SYSOP-05 (USERS keybind gating + named from→to copy):

1. Add `Foglet.Accounts.valid_status_transitions/1` (D-14) — public, derived
   from the existing private `permit_status_transition/2` clauses at
   `accounts.ex:187-191`.
2. Wire `[R] Retry` advertising and dispatch into Sysop's command bar
   (D-13): only when the active tab is in `{:error, _}` with reason ≠
   `:forbidden`. Pressing `R` re-dispatches the matching `{:load_sysop_*}`
   tuple. `:forbidden` suppresses both the hint and the keypress.
3. Convert `UsersView.@footer` (constant string at line 23, 100) into a
   render-time function that consults `Accounts.valid_status_transitions/1`
   for the focused row's current status. Apply the A2 disambiguation rule:
   `[A] Approve` only on `:pending` rows; `[U] Reactivate` only on
   `:suspended` rows.
4. Replace `error_message(:invalid_transition)` (line 213) with a
   from→to copy builder that names the user handle, source status, and
   target status. Output never contains `"invalid_transition"`.

Purpose: A real operator pressing `R` on a healthy tab does not silently
reload it; pressing `R` on a `:forbidden` tab does not loop. Pressing
`A` on an already-`:active` user is a no-op (the keybind is not
advertised). When a transition does fail at the boundary, the user sees
`Cannot change @alice from active to pending.` instead of the raw atom.

Output: `accounts.ex` gains a public predicate; `sysop.ex` gains
`maybe_add_retry/2` + `R` keypress handler; `users_view.ex` swaps the
constant footer for a render-time function and rewrites the error
message; tests cover all four code paths plus the regression that
`"invalid_transition"` never appears in output.
</objective>
