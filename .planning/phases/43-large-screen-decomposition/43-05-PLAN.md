</objective>

<must_haves>
<truths>
- TUI-05: Maintainers can identify each decomposed large screen's reducer-facing module, sibling state owner, and sibling render entry point.
- D-01: All six named render entry points use the `Foglet.TUI.Screens.<Screen>.Render` namespace and `screens/<screen>/render.ex` files.
- D-02: Top-level screen modules remain the reducer-facing `Foglet.TUI.Screen` implementations.
- D-03: Top-level screen modules no longer contain detailed private screen-body render helper families after extraction.
- D-04: Render modules own frame/content/keybar assembly and detailed body/tab/helper rendering.
- D-05: Render modules remain pure over loaded state/context and avoid Repo, PubSub, task starts, durable writes, and screen-state mutations.
- D-06: Render modules keep using `Foglet.TUI.Theme` and existing widgets.
- D-11: Do not introduce reducer modules unless clearly needed; this phase targets state/render/reducer clarity, not abstraction for its own sake.
- D-12: Reducer/effect behavior remains tested through `init/1`, `update/3`, task results, modal submit messages, route entry, or public non-render seams.
- D-13: Render verification covers all six named screens through layout smoke or `rtk mix foglet.tui.render` evidence.
- D-14: Render-module tests assert structural behavior, widget/model routing, or width-safe shape; no snapshot or text-presence-only tests.
- D-15: TUI documentation states when large screens should use sibling `state.ex` and `render.ex`, what top-level screen modules retain, and what render modules must not do.
</truths>
</must_haves>

<action>
Update `lib/foglet_bbs/tui/SCREEN_CONTRACT.md` with a section named exactly `## Large Screen Decomposition`.

The section must state these concrete rules:

- Large screens with substantial reducer and render helper code should use a top-level reducer-facing screen module, sibling `state.ex`, and sibling `render.ex`.
- Top-level screen modules retain `init/1`, `update/3`, optional `subscriptions/2`, effect creation, task result handling, modal submit handling, and public non-render test seams.
- Sibling render modules own frame/content/keybar assembly and detailed body/tab/helper rendering.
- Render modules must not query the database, mutate screen state, subscribe to topics, start tasks, or perform durable domain writes.
- Render modules consume already-loaded screen state plus `Foglet.TUI.Context` or a derived render model.
- Current Phase 43 examples are `PostReader`, `Sysop`, `Login`, `MainMenu`, `NewThread`, and `Account`.
</action>
