<objective>
Extract sibling render modules for Sysop and Account tab shells while preserving existing surface modules, reducer task routing, invite/account/sysop actions, and visibility logic.
</objective>

<must_haves>
<truths>
- TUI-05: Sysop and Account must expose clear reducer/state/render boundaries for large tabbed TUI screens.
- D-01: Render entry points are `Foglet.TUI.Screens.Sysop.Render` and `Foglet.TUI.Screens.Account.Render`.
- D-02: Top-level `Sysop` and `Account` modules keep reducers, task effects, modal/submit handling, tab update routing, and public non-render seams.
- D-03: Top-level render callbacks delegate to sibling render entry points and no longer contain detailed tab/frame helper families.
- D-04: Render modules own frame/content/keybar assembly and detailed tab body dispatch.
- D-05: Render modules must not perform Repo calls, PubSub subscription changes, task starts, durable domain writes, or screen-state mutations.
- D-06: Styling remains routed through `Foglet.TUI.Theme` and existing widgets.
- D-08: Sysop and Account render modules orchestrate existing surface modules instead of replacing `BoardsView`, `UsersView`, `ProfileForm`, `PrefsForm`, `SSHKeysSurface`, and shared invite surfaces.
- D-10: Sysop and Account can be implemented in one wave after lower-risk extractions because they share tab shell patterns but write disjoint render modules.
- D-12: Preserve reducer/effect tests through direct `Sysop.update/3` and `Account.update/3`.
- D-14: Add only structural render/decomposition tests, not text-presence-only tests.
</truths>
</must_haves>
