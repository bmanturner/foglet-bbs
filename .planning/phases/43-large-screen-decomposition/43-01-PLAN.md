<objective>
Extract pure sibling render entry points for NewThread and Login while preserving their reducer-facing screen modules, state owners, input handling, validation, and task effects.
</objective>

<must_haves>
<truths>
- TUI-05: Maintainer can work on the largest TUI screens through separated reducer/state/render modules where mixed responsibilities were identified.
- D-01: NewThread and Login render entry points live at `Foglet.TUI.Screens.NewThread.Render` and `Foglet.TUI.Screens.Login.Render`.
- D-02: `NewThread` and `Login` top-level modules retain `init/1`, `update/3`, task effect creation, task result handling, and validation flow.
- D-03: Top-level `render/2` callbacks delegate to sibling render entry points and no longer contain detailed screen-body helper families.
- D-04: Render modules own frame/content/keybar assembly and detailed form/menu/composer rendering.
- D-05: Render modules must not perform Repo calls, PubSub subscription changes, task starts, durable domain writes, or screen-state mutations.
- D-06: Styling remains routed through `Foglet.TUI.Theme` and existing widgets; no visual redesign or product workflow changes.
- D-09: Login menu/form/reset rendering and NewThread board-picker/composer rendering move to render modules while input handling and validation stay in reducer/state code.
- D-10: Start with smaller/self-contained screens to limit blast radius.
- D-12: Preserve and extend reducer/effect tests around behavior touched by extraction through `Screen.update/3`.
- D-14: Render-module tests must assert structural behavior or module routing, not text-presence-only snapshots.
</truths>
</must_haves>
