Output: Invite-only registration enforcement, atomic invite consumption, and a safe non-consuming register-screen preflight.
</objective>

  <behavior>
    - When `Foglet.Config.registration_mode() != "invite_only"`, `register_user/1` preserves existing open registration behavior and ignores `invite_code`.
    - When mode is `"invite_only"`, missing, unknown, revoked, consumed, or race-lost invite codes return `{:error, changeset}` with a generic error on `:invite_code`.
    - Invalid user attrs are validated before invite consumption and do not mutate the invite row.
    - Successful registration inserts the user and conditionally consumes the invite in one DB transaction; after commit, default board subscription still runs as before.
  </behavior>
  <action>
    Refactor `Foglet.Accounts.register_user/1` to branch on `Foglet.Config.registration_mode()`. For non-`"invite_only"`, keep the current flow: `User.registration_changeset(%User{}, attrs) |> Repo.insert()`, then `Foglet.Boards.subscribe_to_defaults(user.id)` after success.

    For `"invite_only"`, implement `register_invite_only_user/1`. Convert attrs to a map that supports atom and string keys; read `invite_code` from either `:invite_code` or `"invite_code"`. Build `user_changeset = User.registration_changeset(%User{}, attrs)` before any invite mutation. If `invite_code` is nil/blank, return `{:error, invite_code_error(user_changeset)}`. If `user_changeset.valid? == false`, return `{:error, user_changeset}` and do not check or consume the invite.

    Use `Ecto.Multi` or `Repo.transact/1` for the successful path. Insert the user, then run a conditional invite consume step using the transaction repo:
    `from i in Invite, where: i.code == ^code and is_nil(i.consumed_at) and is_nil(i.revoked_at)`.
    Update `consumed_at`, `consumed_by_user_id`, and `updated_at` to `DateTime.utc_now() |> DateTime.truncate(:microsecond)`. Treat update count `1` as success; update count `0` rolls back with `:invalid_invite_code`. On rollback for invalid invite, return `{:error, invite_code_error(user_changeset)}` with the same generic message for unknown, revoked, consumed, and race-lost cases. On insert changeset failure from the Multi, return `{:error, changeset}`. After successful transaction, run `Foglet.Boards.subscribe_to_defaults(user.id)` and return `{:ok, user}`.

    Implement `invite_code_error/1` with `Ecto.Changeset.add_error(changeset, :invite_code, "is invalid or unavailable")`. Do not expose whether the code was unknown, revoked, or consumed.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Guest registration input -> Accounts | Untrusted invite code, handle, email, and password cross into domain registration. |
| Accounts transaction -> database | User creation and invite consumption must commit or roll back together. |
| TUI register screen -> Accounts | TUI is a collector only; it must not consume invite codes or become the trust boundary. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Spoofing | invite-code validation | mitigate | Registration accepts only persisted available codes generated by Plan 02; TUI format preflight is advisory only. |
| T-03-02 | Tampering | `register_invite_only_user/1` | mitigate | User insert and conditional invite consume run in one DB transaction; update count `0` rolls back user creation. |
| T-03-03 | Elevation of privilege | invite consumption path | mitigate | Consumption happens only through `Accounts.register_user/1`; generation/revocation remain protected by Plan 02 domain authorization. |
| T-03-04 | Information disclosure | invite registration errors | mitigate | `invite_code_error/1` returns one generic `invite_code` changeset error for missing, unknown, revoked, consumed, and race-lost codes. |
</threat_model>
