# Plan 02 — Seed max_thread_title_length Config Key

<objective>
Add a new `Foglet.Config` key, `max_thread_title_length`, with a default value of **60**, to `priv/repo/seeds.exs`. This key is consumed by the NewThread title input handler in Plan 04-03 to enforce a soft TUI cap on thread titles (D-13, D-14). The schema's changeset keeps `validate_length(:title, max: 300)` as a hard DB backstop (D-15), so sysops can raise the TUI cap up to 300 via `mix foglet.config.set max_thread_title_length N` without a schema migration.

The seeded value is persisted to the `config_entries` table via `Foglet.Config.put!/3`; on a fresh dev or test DB that hasn't run seeds, callers of `Foglet.Config.get!("max_thread_title_length")` would raise `Ecto.NoResultsError`. Plan 04-03's NewThread code wraps `Config.get!/1` in a safe fallback to 60 so that missing config never crashes the TUI. This plan only seeds the value; the callers are wired in Plan 04-03.

Why 60 rather than 300: user-stated constraint — thread-list rows must fit title + metadata (creator, time-ago, post count from Phase 3) within 80-col terminals. Titles over 60 chars routinely trigger the `…` ellipsis truncation added in Phase 3 D-03. Sixty is the preferred soft cap; the 300-char changeset remains the hard cap so no schema migration is needed if a sysop chooses to loosen the config value later.

This plan follows the exact pattern of the existing `max_post_length` entry in the `default_config` tuple list at `priv/repo/seeds.exs:42-50`. No new code paths are introduced — only one tuple entry in the list.
</objective>

```elixir
defmodule Foglet.ConfigSeedTest do
  @moduledoc """
  Confirms that the seed script inserts config entries consumed by the TUI.

  The seed logic lives in `priv/repo/seeds.exs` — this test exercises
  `Foglet.Config.put!/3` + `Foglet.Config.get!/1` directly to simulate
  what the seed does, without requiring `mix run priv/repo/seeds.exs`
  to succeed in the test env (which would also create users, boards,
  and threads — not relevant here).
  """
  use Foglet.DataCase, async: false

  alias Foglet.Config

  describe "max_thread_title_length (D-13, Phase 4)" do
    test "can be put! and then read via get!" do
      # Simulates the seed tuple {"max_thread_title_length", 60, "..."}
      # from priv/repo/seeds.exs. The seed script calls Config.put!/3 in
      # an Enum.each loop; we test the round-trip for the one new key.
      Config.put!("max_thread_title_length", 60, nil)

      assert Config.get!("max_thread_title_length") == 60
    end

    test "returns 60 as the seeded default when put! is called without a value change" do
      # The seed script is idempotent — running twice doesn't change the
      # value. put!/3 is an upsert (INSERT ... ON CONFLICT UPDATE).
      Config.put!("max_thread_title_length", 60, nil)
      Config.put!("max_thread_title_length", 60, nil)

      assert Config.get!("max_thread_title_length") == 60
    end

    test "a sysop can raise the cap without a migration (D-15 backstop still applies)" do
      # This proves the soft-cap design: the config value can go up to
      # 300 (the schema's validate_length hard cap). The test doesn't
      # exercise the changeset itself — it exercises the config path.
      Config.put!("max_thread_title_length", 300, nil)

      assert Config.get!("max_thread_title_length") == 300
    end
  end
end
```
