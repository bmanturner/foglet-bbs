# Plan 04 — Composer Preview via MarkdownBody + Dead-Code Cleanup

<objective>
Collapse the last of the composer duplication: both `PostComposer` and `NewThread` currently carry their own `render_markdown_tuples/2` (private, line 291-302 and line 208-219 respectively), their own `translate_key/1` (line 180-201 and line 428-450), and their own `render_input_as_text/3` / `render_body/3` (line 246-272 and line 164-192). Plans 04-01 created `Foglet.TUI.Widgets.Compose` and Plan 04-03 left the private copies intact (so the 04-03 behavioral changes could land without refactor risk). This plan finishes the cleanup:

1. **Markdown preview → `Foglet.TUI.Widgets.Post.MarkdownBody.render/3`** (D-11). Both composers' preview modes currently delegate to the private `render_markdown_tuples/2` copies that emit `text/2` nodes from a flat `[{text, style}]` tuple list. This is the exact rendering flaw RENDER-01 fixed in Phase 2 — posts rendered with line breaks as visible `\n` characters instead of sibling layout separation. The fix is to delegate to `Post.MarkdownBody.render/3` (which takes the raw markdown string, width, theme) directly and delete the copies.
2. **Shared `translate_key/1` → `Foglet.TUI.Widgets.Compose.translate_key/1`** (D-09). Delete the private copies in both screens; replace the call sites in `handle_key/2` (PostComposer line 83) and `handle_compose_key/3` (NewThread line 335) with the module-qualified call `Compose.translate_key(key_event)`.
3. **Shared `render_input` → `Foglet.TUI.Widgets.Compose.render_input/4`** (D-09). Delete the private copies in both screens; replace the call sites in `render_input/3` (PostComposer line 239) and `render_body_section/3` (NewThread line 155) with `Compose.render_input(input_st, focused?, theme, opts)` — passing `empty_line_placeholder: " "` in NewThread's call to preserve its legacy placeholder behavior.

This plan is **pure refactor** — zero behavioral change. After landing, the following should all still be true:

- `mix test` passes with the same pass count as before the plan.
- Preview rendering in both composers shows correctly-formatted markdown (headers, bold, italic, lists, etc.) — matching Phase 2's RENDER-01 fix.
- `translate_key/1` still returns the same output for the same input.
- `render_input` still renders cursor, content, and multi-line bodies identically.

**Why this plan is separate from 04-03:** The behavioral changes in 04-03 need their own independently-testable diff. This plan is a mechanical extract-and-delegate that could regress rendering if the MarkdownBody signature is misused. Keeping it separate means a green 04-03 test suite confirms composer behavior; a green 04-04 test suite confirms the refactor didn't regress rendering. If 04-04 is skipped for scope pressure, 04-03 still ships a working Phase 4 — the duplicate code is ugly but functional.
</objective>
