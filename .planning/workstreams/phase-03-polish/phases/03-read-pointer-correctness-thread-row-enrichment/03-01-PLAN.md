# Plan 01 — Boards.advance_board_read_pointer/3: GREATEST Monotonicity Fix

<objective>
Fix the LIST-01 root cause in `Foglet.Boards.advance_board_read_pointer/3` so the board read pointer only ever advances — reading an older thread after a newer one must NOT regress the pointer. Current code at `lib/foglet_bbs/boards.ex:186-198` uses `on_conflict: [set: [last_read_message_number: ^message_number, ...]]`, which overwrites unconditionally. The fix swaps the set expression for a `GREATEST(existing, incoming)` fragment so the pointer is monotonic at the database level. `last_read_at` is updated to `now` unconditionally (it's a "most recent activity" timestamp, not the pointer's identity).

**Why a DB-level fragment and not an Elixir read-compare-write:** A read-then-write sequence is a race in the presence of concurrent advance calls from the TUI `{:flush_read_pointers, ctx}` task and any future PubSub-driven advances. `GREATEST` inside the `on_conflict` runs as a single atomic upsert — no lost update is possible. This matches Ecto's documented pattern for conditional upserts and keeps the call site unchanged.

**Contract preserved:** The function signature, return shape (`{:ok, ReadPointer.t()} | {:error, Changeset.t()}`), and the `unique_constraint [:user_id, :board_id]` stay the same. Only the `on_conflict` behavior tightens from "replace" to "GREATEST(existing, incoming)". The initial-insert branch is unaffected — a new row still gets `last_read_message_number: ^message_number`.

**Existing test at `test/foglet_bbs/boards/boards_test.exs:126-145` ("updates existing pointer on subsequent call (upsert)") currently asserts `ptr.last_read_message_number == 7` after an advance from 3 → 7.** That test still passes after this change (7 > 3). A new test must assert the reverse sequence (3 after 7 stays at 7). Without that test, the bug can silently return.
</objective>

Replace the entire function body with:

```elixir
  @doc """
  Advance (or create) a board read pointer (LIST-01 monotonic).

  The `last_read_message_number` only ever increases — the on_conflict
  upsert uses a `GREATEST(existing, incoming)` fragment so that reading
  an older thread after a newer one does NOT regress the pointer.
  `last_read_at` is updated to `now` unconditionally because it
  represents "most recent activity", not the pointer's identity.

  Safe to call multiple times with the same or lower message_number —
  the pointer stays at the max value seen so far.
  """
  @spec advance_board_read_pointer(String.t(), String.t(), integer()) ::
          {:ok, ReadPointer.t()} | {:error, Ecto.Changeset.t()}
  def advance_board_read_pointer(user_id, board_id, message_number)
      when is_integer(message_number) and message_number >= 0 do
    now = DateTime.utc_now()

    %ReadPointer{user_id: user_id, board_id: board_id}
    |> ReadPointer.changeset(%{
      last_read_message_number: message_number,
      last_read_at: now
    })
    |> Repo.insert(
      on_conflict: [
        set: [
          last_read_message_number:
            fragment(
              "GREATEST(board_read_pointers.last_read_message_number, ?)",
              ^message_number
            ),
          last_read_at: ^now
        ]
      ],
      conflict_target: [:user_id, :board_id],
      returning: true
    )
  end
```

**Why `returning: true`:** Without it, Ecto returns the stale struct the caller constructed (with the incoming message_number), not the actual row — so the caller sees `ptr.last_read_message_number == 3` even though the DB row stayed at 7. `returning: true` forces Ecto to SELECT the row after the upsert so the caller gets the real, monotonic value.

**Why `fragment/1` with `^message_number` interpolation:** Ecto's query-expression fragments must use `^` for parameter interpolation so Postgres sees a prepared-statement parameter (prevents SQL injection; enables query plan caching). The literal `board_read_pointers.last_read_message_number` on the left side of the `GREATEST` references the existing row's column — Postgres resolves this automatically inside an ON CONFLICT clause.

**Why the function head guard `when is_integer(message_number) and message_number >= 0`:** Guards against an accidental `nil` or negative message_number (which would make the GREATEST comparison asymmetric and surprising). Matches the schema default of `0` and the `integer` column type.

      {:ok, first} = Foglet.Boards.advance_board_read_pointer(user.id, board.id, 7)
