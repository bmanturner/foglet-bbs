# Plan 04 — Screen Integration: Thread Row Enrichment + Read-on-Entry + Two-Phase Board Refresh

<objective>
Wire the three Wave 1 artifacts (`Boards.advance_board_read_pointer/3` GREATEST fix, `Threads.list_threads/2` annotation, `ListRow.render_with_metadata/6`) into the TUI flow so the phase goals land end-to-end:

1. **LIST-01 read-on-entry (D-05):** PostReader's `load_posts/2` seeds `state.read_position[thread_id]` with post 0's `{post_id, message_number}` **before** the user navigates. Pressing Q immediately after entry advances the board pointer past the first post — the existing `{:flush_read_pointers, ctx}` path is already correct once `read_position` is populated on entry; this plan makes that population happen.

2. **LIST-02 two-phase board refresh (D-06):** When the user presses Q from ThreadList, ThreadList's handle_key dispatches `[{:load_boards}]` in addition to changing the screen. When `App.do_update({:read_pointers_flushed, thread_id}, state)` fires and `state.current_screen == :board_list`, App dispatches a second `{:load_boards}` task to pick up the post-flush counts.

3. **LIST-03 thread row enrichment (D-01..D-04):** ThreadList's `render_thread_content/3` loads threads via `Threads.list_threads/2` (passing the user_id), formats the metadata string as `"@handle · N posts · Xh ago"`, and calls `ListRow.render_with_metadata/6` with the appropriate `unread?` flag and current terminal width. The data path in App's `{:load_threads, board_id}` task also switches to the 2-arity call so PubSub-driven thread refreshes get the same annotation.

**The PostReader `:post_reader` screen_state shape already exists from Phase 2 Plan 03** (`%{selected_post_index: _, scroll_offset: _, render_cache: _}`); this plan does NOT touch that shape. It only modifies `load_posts/2` to seed `read_position` on entry.

**Why all three integrations in a single plan:** They share a testing surface (ThreadList screen tests, PostReader screen tests, App update-dispatch tests) and a scope boundary (the screen/app layer). Splitting them across plans would force the test files to be touched three times and increase the risk of a sub-plan landing in isolation that leaves the acceptance criteria in a half-finished state. Wave-2 grouping matches the Phase 2 template where PostReader integration was also a single plan consuming all Wave-1 outputs.

**Scope fence — phase 2 untouchables:** This plan does NOT modify `lib/foglet_bbs/tui/widgets/post/markdown_body.ex`, `lib/foglet_bbs/tui/widgets/post/post_card.ex`, or any other Phase 2 widget surface. CONTEXT.md "Known Stubs" explicitly states "Phase 3 must not touch them." PostReader receives a single-line change to `load_posts/2` that seeds `read_position` — no render-path changes.

**Metadata formatting (resolved D-02):** String template is `"@#{handle} · #{post_count} posts · #{time_ago} ago"`. Singular "post" is used when `post_count == 1`. When the handle is missing, `"@unknown"` is the fallback. When `last_post_at` is nil (empty thread edge case), the time-ago segment collapses to `"new"` (no trailing " ago"). Per CONTEXT D-02 all three pieces are always separated by ` · ` (U+00B7 middle dot).
</objective>
