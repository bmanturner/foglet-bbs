# Plan 01 — Config Seeds + `Foglet.Accounts.post_login_screen/1`

<objective>
Lay the VERIFY-01 foundation: seed two new `Foglet.Config` keys and add a single, narrowly scoped Accounts function `post_login_screen/1` that returns `:main_menu | :verify` based on the user's `confirmed_at` and the `require_email_verification` config flag. This plan touches ONLY `priv/repo/seeds.exs`, `lib/foglet_bbs/accounts.ex`, and `test/foglet_bbs/accounts/accounts_test.exs`. No TUI screen changes — those land in Plan 06-02.

**Why this split:** The Accounts function is pure business logic, easily unit-tested in isolation. Plan 06-02 swaps the two TUI call sites to use it. Plan 06-03 then extends the verify screen's cooldown state. The three plans are sequenced by dependency — each wave proves its contract before the next consumer wave starts.

**Contract:**
- `Foglet.Accounts.post_login_screen/1` takes a `%User{}` and returns `:main_menu | :verify` (D-04).
  - Confirmed user (`user.confirmed_at != nil`) → `:main_menu` — regardless of the config flag.
  - Unconfirmed user (`user.confirmed_at == nil`) AND `require_email_verification == false` → `:main_menu`.
  - Unconfirmed user AND `require_email_verification == true` (or missing key) → `:verify`.
- Config reads use `Foglet.Config.get/2` with default `true` so missing seeds (stale test DB) don't raise.
- Two new seed entries appended to the `default_config` tuple list in `priv/repo/seeds.exs` at line 43-49. Idempotent via the existing `Enum.each` loop (already checks `Repo.get_by(Entry, key: key)`).

**Non-goals for this plan:**
- No changes to `login.ex`, `register.ex`, `verify.ex`, or `app.ex`.
- No changes to the `Foglet.Config` module itself — its API is stable.
- No seed migration for existing `confirmed_at: nil` users (D-07 locked: retroactive bypass is login-guard-only).
- No config admin TUI or Mix task — sysops use `Foglet.Config.put!/3` from IEx (deferred to Milestone 8).
</objective>

Immediately after the closing `end` of `confirm_user/1` (line 153) and BEFORE the `@doc` block at line 155, insert this function:

```elixir

  @doc """
  Decide which screen the user lands on after a successful login or
  registration (Phase 6 D-04, VERIFY-01).

  Returns:
    * `:main_menu` — user is confirmed (`confirmed_at != nil`), OR
                    `require_email_verification` is disabled globally.
    * `:verify`   — user is unconfirmed AND verification is required.

  Reads the `require_email_verification` config key with a safe default of
  `true` — a missing seed entry (stale test DB) defaults to "verification
  required" rather than silently bypassing the check.

  Retroactive bypass policy (REQUIREMENTS locked): an existing user with
  `confirmed_at: nil` gains access on their next login when the sysop flips
  the toggle to `false`; no DB migration is performed.
  """
  @spec post_login_screen(User.t()) :: :main_menu | :verify
  def post_login_screen(%User{confirmed_at: confirmed_at}) do
    cond do
      confirmed_at != nil ->
        :main_menu

      Foglet.Config.get("require_email_verification", true) == false ->
        :main_menu

      true ->
        :verify
    end
  end
```

**Why `cond` instead of nested `if`:** Three distinct branches with equal priority read more clearly as a `cond` than as nested `if`s. The order is intentional: confirmed-check first (cheapest, no config read), then config read, then the default.

**Why `Foglet.Config.get/2` (not `get!/1`):** A missing config row would raise in `get!/1`, breaking test suites that skip seeds. `get/2` with default `true` preserves the "verification required" posture when the key is absent — the safer failure mode.

**Why `%User{confirmed_at: confirmed_at}` destructure:** Avoids a second `user.confirmed_at` lookup and makes the function's dependency on one field obvious. Matches the `%User{} = user` idiom used elsewhere in the file (e.g. `register_user/1` uses `%User{}` at line 45).

**Why explicit `== false` (not `!=`):** Readable as "when the sysop has disabled verification." `Foglet.Config.get("...", true)` returns `true | false` for this key — no other shapes.
</action>
