# Plan 03 — Verify Screen: Independent Resend Cooldown (VERIFY-02)

<objective>
Split the single `cooldown_until` field on `verify_state` into two independent timers: `cooldown_until` (invalid-attempts anti-brute-force cooldown, UNCHANGED behavior) and the new `resend_cooldown_until` (anti-spam timer for the R key). Introduce a `resend_cooldown?/1` helper and a `cooldown_modal/2` variant that takes an explicit DateTime + message prefix so the two cooldowns can report independent feedback. Read the resend duration from `Foglet.Config.get("email_verify_resend_cooldown_seconds", 60)` so sysops can tune it without a code change.

**Why two timers, not one:** Per CONTEXT.md D-10, "invalid attempts = lock me out temporarily; resend = prevent me from spamming." These are different anti-abuse concerns that users reason about separately. Sharing `cooldown_until` meant hitting invalid 5 times blocked resend (wrong), and hitting resend once blocked code entry (wrong). Two timers, two behaviors, two UX messages.

**Why the config-driven resend duration:** Per D-02, the resend cooldown is a polish-phase knob sysops will want to tune without redeploying. The invalid-attempts cooldown stays as a module attribute `@cooldown_seconds 60` (D-03 scope discipline — not every knob needs config).

**Contract:**
- `verify_state` gains a new key `resend_cooldown_until :: DateTime.t() | nil`. Every initialisation site (6 in `verify.ex`) adds `resend_cooldown_until: nil`.
- `resend_cooldown?(vs)` private helper mirrors the existing `cooldown?/1`: returns true when `vs.resend_cooldown_until` is in the future.
- `cooldown_modal(datetime, prefix)` replaces the zero-argument `cooldown_modal/1`. Callers pass an explicit DateTime and a prefix message.
- `resend_code_raw/1`:
  - Before generating: check `resend_cooldown?(vs)`. If active, return `%{type: :error, message: "Please wait <N>s before requesting another code."}` and DO NOT mutate state beyond the modal.
  - After successful generation: set `resend_cooldown_until: DateTime.add(now, cooldown_seconds, :second)` where `cooldown_seconds = Foglet.Config.get("email_verify_resend_cooldown_seconds", 60)`.
  - Continue to clear `buffer: ""`, `attempts: 0`, `cooldown_until: nil` on success (D-09 locked — matches current behavior).
- `resend_code/1` (public entry from R key): now checks `resend_cooldown?/1` instead of `cooldown?/1`.
- `handle_key/2` typed-char branch keeps checking `cooldown?/1` (invalid-attempts cooldown, UNCHANGED) — hit invalid 5x → can still resend; hit resend once → can still enter codes.

**Non-goals:**
- No change to `@max_attempts`, `@cooldown_seconds` (invalid-attempts), or `@code_length` module attrs (D-03 locked).
- No modal / Widgets.Compose refactor — `cooldown_modal/2` stays private in `verify.ex`.
- No session-level invalidation when the config flips mid-session (D-14, D-15 locked).
</objective>

**Why `cooldown_until: nil` stays in the set expression (D-09):** The locked decision says successful resend clears the invalid-attempts cooldown. This mirrors "user's code landed, clean slate" intuition. Plan 06-03 preserves this — the two cooldowns are independent BUT a successful resend is an explicit reset of the invalid-attempts state.
